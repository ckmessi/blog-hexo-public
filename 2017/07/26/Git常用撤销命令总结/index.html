Git常用撤销命令总结

2017-07-26

在使用Git时，很多时候我们会遇到需要相关的撤销命令。这有可能是因为误执行了git add或git commit操作，也有可能是之前执行了相关操作后，发现内容并不是我们真正想要的。然而Git在不同状态下所需要执行的撤销命令是不同的，容易混淆，这里对常用的撤销命令进行总结。

原理分析

在git仓库中，通常文件会有5种状态（不考虑未untracked files）的情况：

  1. 未修改，与仓库内容保持一致
  2. 修改、但未执行git add。（Changes not staged for commit）
  3. 修改、并执行了git add，但未执行git commit。（Changes to be committed）
  4. 修改、并执行了git add & git commit，但未执行git push推送到远程。
  5. 修改、并执行了git add & git commit & git push

其中状态0、1的文件位于工作区，状态2的文件位于暂存区，状态3、4的文件已经被提交到了仓库中。

接下来分析处于各种状态下的撤销命令。

状态3的撤销

对于状态3，是指修改了版本库中的内容，并执行了git add & git commit，此时进行撤销操作目的是恢复到上一个commit版本（或上某个commit版本），同时修改的内容还存在（或消失）。

git reset命令实质上是重设版本，HEAD^表示上一个版本，commit_id可以在git log中看到。
--hard参数表示，将重设目标commit_id后工作区的所有改变都丢弃，并把HEAD指向commit_id。这个模式的效果是，所有修改都丢弃，代码回滚到commit_id版本，且工作区是干净的。
--soft参数表示，不重设Index和工作区，仅仅把HEAD指向commit_id。这个模式的效果是，从commit_id之后的修改都会显示在Changes to be committed（即状态2）中。
--mixed参数表示，仅重设Index，不重设工作区。这个模式的效果是，从commit_id之后的修改会显示在Changes not staged for commit（即状态1）中。
总结一下：
从状态3=>状态2，使用git reset --soft HEAD^
从状态3=>状态1，使用git reset --mixed HEAD^
从状态3=>状态0，使用git reset --hard HEAD^

状态4的撤销

对于已经执行git add & git commit & git push的操作，想要撤销其实是非常困难的。别人还没有来得及pull这个分支。
在别人还没有来得及pull这个分支的前提下，解决方法是使用状态3的撤销，并将最新内容再次推送到远程分支。

总结

在不考虑远程分支及合并等情况下，本地工作Git目录中的内容可能处于5种状态，针对这5种状态进行撤销操作的命令如下：

  • 状态0（未修改）：不需要撤销
  • 状态1（修改）：git checkout -- [filename]
  • 状态2（修改，并git add）：git reset HEAD [filename]
  • 状态3（修改，并git add & git commit）：git reset --hard|--mixed|--soft HEAD^ [filename]
  • 状态4（修改，并git add & git commit & git push）：参考状态3进行撤销，并执行git push --force
