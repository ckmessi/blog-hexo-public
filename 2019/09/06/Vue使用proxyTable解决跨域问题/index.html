Vue使用proxyTable解决跨域问题

背景

之前在使用 Vue.js 作为前端框架时，通常是在主方法中引入 axios ，并设置其 baseURL 属性指向后端：

在这种情况下，如果前端页面所在的域名和端口号与后台服务器不同，需要后台服务器提供支持，因为涉及到了跨域请求。

还有一种方式，可以在 Vue.js 开发过程中直接避免跨域问题，就是使用配置中的 proxyTable 项。

解决方案

target 属性标识了真实的后台域名，changeOrigin 设置为 true 可以允许跨域，而 pathRewrite 则标识了一个重写逻辑，这部分是可选的。

这样，如果前端页面所在的域名端口是 http://127.0.0.1:8888 ，那么 ExecuteSingle 方法执行时发出的请求就是 http://127.0.0.1:8888/backend-api/web/academy/list 。
此时这个请求将被转发到 proxyTable 中 target 属性 http://localhost:8710/ 处，即实际执行的请求会转变成 http://localhost:8710/backend-api/web/academy/list ；同时，由于 pathRewrite 属性的声明，其中路径中的 ^/backend-api 部分还会被重写为 xxx，因此最终实际的请求是 http://localhost:8710/xxx/web/academy/list。

注意，上述使用场景都是针对 npm run dev 启动时才有效。
注意：修改 config/index.js 后，一定要重新运行 npm run dev 才会生效！！！

配置到 nginx

上述解决方法都是针对使用 npm run dev 场景下才有效，真正部署至线上时，我们通常会使用 npm run build 打出一个 dist 包，并使用 Nginx 服务器进行部署。
此时 dist 目录中的前端代码是不受上述代理配置约束的，它会按照正常的逻辑访问 http://127.0.0.1:8888/backend-api/web/academy/list 。

但在实际使用中发现，经过一次 proxy_pass 的请求，偶尔会出现一直 pending 的情况，直到超时了才重试，重试后又正常返回。

具体现象和 这篇文章 描述的很像，排查了好长时间，最终根据 这篇博客 知道了原因。
标准解决方法是将 localhost 修改为 127.0.0.1 即可，妥协的解决方法可以是设置一个较少的 proxy_connect_timeout 值让其超时自动重试。
