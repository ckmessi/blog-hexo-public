Git克隆项目时SSH/HTTP方式的区别

在使用Git获取仓库中的项目时，通常可以选择用SSH或HTTP方式。
两种克隆方式对应的URL也不一样。以Github为例，选择Clone with SSH时，对应的URL为：

git@github.com:ckmessi/ckmessi.github.io.git
而选择Clone with HTTPS时，对应的URL为：

https://github.com/ckmessi/ckmessi.github.io.git
两种方式的区别在于：

使用SSH方式时，需要预先配置ssh key相关信息，克隆过程实质上建立了一个SSH连接（就像远程免密登录服务器一样）；如果在配置ssh key时没有设置密码，则每次push时不需要验证密码。
使用HTTP方式时，可以不经任何验证克隆该项目的数据，但在每次push的时候，都需要验证用户名和密码。
类似于SSH是一项永久性操作，而HTTP只是快速构建临时可读的方案。

因此，在实际工作中，如果要频繁地获取并提交修改，应该使用SSH方式；如果只是将项目克隆下来随意浏览，不需要提交修改操作，则使用HTTP更方便。

使用Guzzle访问API接口需要登录的解决方案

最近在工作中遇到一个问题，需要撰写脚本模型HTTP请求进行一些批量操作。而这些HTTP请求是需要登录状态的。
在这种情况下，使用 Guzzle 连续发送两条请求，并不能模拟它们在相同浏览器下操作的效果。
因此，不能直接以 请求登录接口 + 请求业务接口 的方式完成。

查看 Guzzle 的文档，根据关于 Cookies 相关的教程可知，在使用 Guzzle 发送请求时，可以让其携带一段 cookie session 。
这样，如果两个接口所携带的 cookie session 相同，自然能达到浏览器下的效果。

Guzzle can maintain a cookie session for you if instructed using the cookies request option. When sending a request, the cookies option must be set to an instance of GuzzleHttp\Cookie\CookieJarInterface.

You can set cookies to true in a client constructor if you would like to use a shared cookie jar for all requests.

这样，通过携带 Cookies 的方式，可以解决 Guzzle 调用API接口需要登录状态的问题。

PHP中警惕悬挂引用可能引起的bug

在PHP中，有一种情况可能由于悬挂引用造成的bug。

如果希望在遍历过程中操作数组的每个元素，可以在 foreach 循环中使用引用。比如：

问题在于，如果不小心的话，可能造成一些意料之外的副作用和不良后果。在上述样例的代码执行之后， $value 变量将停留在当前作用域中，保持着对数组最后一个元素的引用。因此，后续有关 $value 变量的操作都可能在不经意间修改了数组最后一个元素的值。

需要记住 foreach 并没有创建一个全新的作用域。上述样例中的 $value 是在脚本代码顶级作用域的一个 引用 。 在 foreach 的每次迭代时，将设置这个引用指向 $array 数组的下一个元素。当循环结束时， $value依然指向着 $array 的最后一个元素，停留在了作用域中。

在经历第一个 foreach 循环时， $array 不曾被改变 ，$value 作为一个悬挂的引用被遗留下来，指向了 $array 数组的最后一个元素。（因为这个 foreach 循环通过引用的方式访问 $value）

结果，当我们经历第二个 foreach 循环时，“奇怪的事情”发生了。由于 $value 现在是通过值访问（即拷贝方式），在循环的每一步， foreach 按顺序拷贝 $array 中的每一个元素到 $value 中。因此，以下是第二个 foreach 循环中每一步所发生的事情：

  • 第1步： 拷贝 $array[0] (这个元素的值是”1”) 到 $value （此时是 $array[2] 的引用）, 因此现在 $array[2] 等于1，而 $array 现在包含的元素是 [1, 2, 1]。
  • 第2步： 拷贝 $array[1] (这个元素的值是”2”) 到 $value （此时还是 $array[2] 的引用），因此现在 $array[2] 等于2，而 $array 现在包含的元素是 [1, 2, 2]。
  • 第3步： 拷贝 $array[2] (这个元素的值现在是”2”) 到 $value （此时依然是 $array[2] 的引用）， 因此现在 $array[2] 依然等于2，而 $array 现在包含的元素是 [1, 2, 2]。

可以选择在http{ }中设置：client_max_body_size 20m;

还可以选择在location{ }中设置：client_max_body_size 20m;

初识AIDL

简介

AIDL（Android Interface Definition Language），称为Android接口定义语言，目的是为了实现Android的进程间通信，尤其是涉及到多进程并发情况下的进程间通信。

宏观上讲，可以将两个需要通信的进程分别视为客户端和服务端，双方定义一份aidl文件，作为通信协议约定。
服务端需要保存xxx.aidl文件，并实现其中定义的方法；客户端需要保存xxx.aidl文件，并以此尝试与服务端建立连接，调用其相关方法。
AIDL的使用过程主要可划分为以下几个步骤：

  • 创建一份aidl文件，在其中定义接口；
  • 将进程间通信的被动端作为服务端，在服务端创建Service，实现aidl文件中的接口所定义的方法逻辑，用于处理客户端的调用请求；
  • 将主动调用端作为客户端，与服务端建立连接，并调用相关方法获取结果。

下面将以一份实现样例进行说明，代码可参考：https://github.com/ckmessi/AIDLDemo/

客户端常用流程概括

在现实场景中，最可能发生的是当前程序需要通过AIDL调用其它进程的某项服务。

项目中需要用到的已知信息如下：

  • 目标服务的包名称，如com.x.example
  • AIDL文件XService.aidl，对应接口名称为IXService

对于客户端来说，所需要进行的流程为以下几点：

  • 将AIDL文件添加到项目中
  • 在项目中定义一个连接XConnection和一个服务XService，使用bindServce方法，将连接与对应的目标服务绑定，将绑定结果存储到XService中
  • 通过获取到的服务XService调用对应的接口
