Azure Application Insights入门

Application Insights是微软Azure平台的一项服务，声称能够全面监控和了解服务的行为、可靠性和性能。
可以这样理解，当一项应用添加了Application Insights插件后，相当于给自己的应用添加了许多监控机制，可以通过Azure平台页面直接监测服务的运行状况。类似于阿里云平台查看云主机吞吐量，或自行使用tail -f x.log命令监控日志信息。

基本概念

首先需要理解以下几个概念：

  • Azure Application Insights：微软推出的服务监控插件，可自行配置安装到应用中。
  • Microsoft Azure：微软推出的云平台，用于监控配置了Application Insights的服务资源。
  • InstrumentationKey：检测密钥，一项Azure上的Application Insights资源对应唯一的密钥，当项目配置密钥后，将直接将该项目应用绑定到密钥所标识的Azure资源上。

Application Insights在ASP.NET中的使用

设置InstrumentationKey

Application Insights的基本使用方法可以参考微软官方文档。
主要包括安装相关SDK，初始化Application Insights配置等流程。
最重要的步骤是添加InstrumentationKey，事实证明所有流程的最终结果本质上是在ApplicationInsights.config文件中添加该配置项，这样应用就能够连上Azure监控平台。

由于不同部署平台可能需要使用不同的InstrumentationKey，比较好的方案是在程序启动后执行的代码中读取配置文件动态添加。
参考文档，可以在项目的启动添加中动态添加InstrumentationKey。

设置cloud_RoleName

cloud_RoleName的作用是区分使用同一个InstrumentationKey的不同组件（Components）。例如，在同一个应用下，可能有多个组件组成，此时如果希望对一些性能指标、日志记录进行筛选，则需要给每个组件添加唯一的cloud_RoleName，在Azure监控平台上的”搜索”中可以添加cloud_RoleName的筛选条件。

参考官方文档1和cloud_RoleName的覆写方法，可以在程序启动时为应用配置特殊的cloud_RoleName。

添加遥测追踪

参考文档可以添加日志记录，最基础的方案是手动添加一项遥测数据。

Solution(.sln) File in Visual Studio

使用Visual Studio创建的项目都包含一个.sln文件，本文简要说明文件的内容及相关作用。

文件说明

在Visual Studio中，解决方案是用于组织管理项目的结构。解决方案将所有项目的状态信息保存在.sln文件和.suo文件中。.sln文件以文本信息的方式，保存着环境需要用于查找和载入项目及其引用VSPackage的持久化数据name-value参数。当一个用户打开解决方案时，开发环境将遍历.sln文件中的preSolution、Project和postSolution信息，载入解决方案、解决方案中的项目、以及解决方案相关联的持久化信息。

内容概览

第一部分，版本信息

第二部分，项目信息

第三部分，全局信息

开发环境载入流程

参考(MSDN官方文档)[https://msdn.microsoft.com/en-us/library/bb165951.aspx]，为了载入`解决方案`，环境平台将执行以下任务：

  1. 开发环境将阅读Global区域，处理所有标记为preSolution的区域，例如：

  2. 开发环境载入VSPackage，调用其中的接口QueryInterface，并使用区域中的数据调用ReadSolutionProps方法，这样数据就能存储在VSPackage中。开发环境将对每个preSolution区域重复这个流程。

  3. 开发环境迭代存留区域（persistence blocks）的项目。

这份声明包含了唯一的项目GUID，以及项目类型GUID。这些信息是被开发环境用于找到属于解决方案的项目文件，以及每个项目所需要的VSPackage。项目GUID将被传递给IVsProjectFactory以载入相关的VSPackage，然后载入项目。

  1. 基于.sln文件项目区域所包含的信息，环境变量载入每个项目文件。项目自身需要负责构建项目层级并载入内嵌的项目

  2. 在.sln文件所有区域后，解决方案将被显示在Solution Explorer中，并准备好呈现给用户。

如果解决方案中任何实现项目的VSPackage载入失败，将调用OnProjectLoadFailure方法，其它项目将让用户选择是否忽略。如果发生解析错误，开发环境将显示对话框警告用户解决方案损坏了。

总结

.sln文件包含了解决方案所需要的相关信息，包括开发环境的版本要求、属于解决方案的项目信息、载入解决方案所需要执行的相关操作等。在启动时，开发环境将根据.sln文件执行相关准备，完成后将解决方案呈现给用户，否则提供相关的提示信息。

Git常用撤销命令总结

在使用Git时，很多时候我们会遇到需要相关的撤销命令。这有可能是因为误执行了git add或git commit操作，也有可能是之前执行了相关操作后，发现内容并不是我们真正想要的。然而Git在不同状态下所需要执行的撤销命令是不同的，容易混淆，这里对常用的撤销命令进行总结。

原理分析

在git仓库中，通常文件会有5种状态（不考虑未untracked files）的情况：

  1. 未修改，与仓库内容保持一致
  2. 修改、但未执行git add。（Changes not staged for commit）
  3. 修改、并执行了git add，但未执行git commit。（Changes to be committed）
  4. 修改、并执行了git add & git commit，但未执行git push推送到远程。
  5. 修改、并执行了git add & git commit & git push

其中状态0、1的文件位于工作区，状态2的文件位于暂存区，状态3、4的文件已经被提交到了仓库中。

接下来分析处于各种状态下的撤销命令。

撤销命令分类

状态0的撤销

对于状态0，不存在需要使用撤销命令的场景。

状态1的撤销

对于状态1，是指修改了版本库中的某些文件内容，希望撤销这些修改，使其恢复到与仓库原内容一致的情况。
注：如果修改的文件不在版本库中（Untracked files），只需要不对这个文件执行git add操作，即不会影响Git仓库的使用。

状态2的撤销

对于状态2，是指修改了版本库中的内容，并执行了git add将其添加到暂存区了，此时进行撤销操作的目的是将其从暂存区移除（即回到没有git add之前的状态）。
这时需要使用的命令是：

若想将修改内容都一起撤销，可参考状态1的情况，执行git reset HEAD [filename] & git checkout -- [filename]

状态3的撤销

对于状态3，是指修改了版本库中的内容，并执行了git add & git commit，此时进行撤销操作目的是恢复到上一个commit版本（或上某个commit版本），同时修改的内容还存在（或消失）。

git reset命令实质上是重设版本，HEAD^表示上一个版本，commit_id可以在git log中看到。
--hard参数表示，将重设目标commit_id后工作区的所有改变都丢弃，并把HEAD指向commit_id。这个模式的效果是，所有修改都丢弃，代码回滚到commit_id版本，且工作区是干净的。
--soft参数表示，不重设Index和工作区，仅仅把HEAD指向commit_id。这个模式的效果是，从commit_id之后的修改都会显示在Changes to be committed（即状态2）中。
--mixed参数表示，仅重设Index，不重设工作区。这个模式的效果是，从commit_id之后的修改会显示在Changes not staged for commit（即状态1）中。
总结一下：
从状态3=>状态2，使用git reset --soft HEAD^
从状态3=>状态1，使用git reset --mixed HEAD^
从状态3=>状态0，使用git reset --hard HEAD^

状态4的撤销

对于已经执行git add & git commit & git push的操作，想要撤销其实是非常困难的。别人还没有来得及pull这个分支。
在别人还没有来得及pull这个分支的前提下，解决方法是使用状态3的撤销，并将最新内容再次推送到远程分支。
如果提示说版本落后于远端版本，需要在push时使用--force参数：

总结

在不考虑远程分支及合并等情况下，本地工作Git目录中的内容可能处于5种状态，针对这5种状态进行撤销操作的命令如下：

  • 状态0（未修改）：不需要撤销
  • 状态1（修改）：git checkout -- [filename]
  • 状态2（修改，并git add）：git reset HEAD [filename]
  • 状态3（修改，并git add & git commit）：git reset --hard|--mixed|--soft HEAD^ [filename]
  • 状态4（修改，并git add & git commit & git push）：参考状态3进行撤销，并执行git push --force
