初识Flyway

背景

对于熟悉 Laravel 框架的朋友们，可能非常喜爱其数据库迁移工具。
对于已经上线的项目，进行数据库结构的改变需要非常小心，使用统一的迁移脚本能够最大程度避免不同人修改数据库却在部署时忘记执行的情况。
类比于 Laravel 的迁移工具，在 Java 中我们可以使用 Flayway 进行数据库迁移。

  • Gradle

一些基础命令

第一步，使用命令 mvn flyway:baseline 会在数据库中创建一个基础的表 flyway_schema_history ，此后 Flyway 进行迁移的所有版本历史都会记录在这个表中。

Migrate

然后可以使用 mvn flyway:migrate 命令，执行后会发现数据库中多了一张 Person 表。
命令 mvn flyway:migrate 本身是幂等的，多次执行的效果是一样的。但如果改动了此前执行过的脚本，再次执行 mvn flyway:migrate 时就会报错。因此所有对数据库的改动，都应该新添加一个脚本而不是修改之前的脚本。

Clean

可以使用 mvn flyway:clean 清除数据库中的所有表信息，使用这个命令前要相当谨慎。

Info

可以使用 mvn info 查看相关信息，其实质效果就是将 flyway_schema_history 表中的内容打印出来。

Validate

命令 mvn flyway:validate 是一个检查操作，它会在以下几种情况发生时失败：

  • 迁移脚本文件的名称、类型或校验和发生变动；
  • 添加了一些版本的脚本文件，但还没有应用；
  • 应用了某些版本的脚本文件，但发生的错误；

比如我们添加一个新的 V2__Create_user_table.sql 文件，不执行迁移操作，执行 mvn flyway:validate 操作，就会发现它失败了。
简单地讲，使用 mvn flyway:validate 操作可以用于检查当前数据库状态是否与迁移脚本库保持一致。

Repair

当迁移执行失败，且数据库不支持 DDL事务 时，迁移将在 schema_hisotry_table 中被标识为 failed 状态，标识了需要手动清除数据库相关内容。
使用 mvn flyway:validate 命令将移除所有失败的迁移，并重新计算检验和等信息。
但要注意的是，一些数据库中遗留的用户对象还需要被手动清除。

Undo

使用这个命令能够撤销上一次迁移操作，然而只有付费的 Flyway Pro 才可以使用这个命令。

不同的迁移类型

在上述样例中，我们创建的迁移脚本名称是 V1_Create_person_table.sql，这实际上包含了一个信息，说明这个迁移是一个 Versioned Migration（基于版本的迁移）。
事实上 Flyway 还有其它类型的迁移：

  • Undo Migrations（撤销迁移）
  • Repeatable Migrations（可重复迁移）

基于版本迁移

其中 基于版本迁移 具有一个 版本号 ，且这个版本号是唯一的，这些迁移按照顺序只会被执行一次。
通常 基于版本迁移 会应用于 创建/修改/删除 数据库中的 表/索引/外键/枚举 等，也可能用于关联数据的更新，用户数据的纠正。大多数情况下，可以按照递增的正整数赋予迁移脚本版本号，但 Flyway 也是很灵活的，可以接受小数点分隔的版本号。
Flyway 会按照版本顺序执行这些迁移脚本，版本号则按照你正常期望的方式排序。

撤销迁移

而 撤销迁移 是正常版本迁移的反向操作，通常用于撤销相同版本迁移的操作效果。撤销迁移的使用是可选的。

可重复迁移

对于 可重复迁移 而言没有版本号的概念，只要它们的内容校检和发生了变化，就会被执行。
通常而言，这类脚本对于管理数据库中的一些定义对象会比较有用，这些内容就可以被一个基于版本控制的单个文件维护起来，具体的例子有创建视图、过程、方法、包等。
在一次迁移执行过程中，可重复迁移 脚本总是在最后执行，在所有等待执行的 基于版本的迁移 脚本完成之后。
可重复迁移 脚本的执行顺序依据其描述内容排序得到。
要使用 可重复迁移 脚本，就必须确保相同的脚本能够被执行多次，比如使用 CREATE OR REPLACE 语句。

Java中BigDecimal的scale和precision

在Java中，经常使用 BigDecimal 类进行高精度的计算。

BigDecimal 类有两项关键的属性：scale 和 precision ，其数值与 BigDecimal 的定义有关。

定义

一个 BigDecimal 对象由以下两个值确定：一个 任意精度整型数 和一个 32位整数表示的尺度 ，它的值按照如下公式定义：

Precision

其中 precision 表示 unscaledValue 中数字位数，比如对于数值 123.45 ，其 precision 就是 5。
因此，precision 指示了任意精度整型数的长度，以下是一些相同 scale 情况下不同 precision 的例子：

  • 0.12345 = 12345 * 10^{-5} // scale = 5, precision = 5
  • 0.12340 = 1234 * 10^{-5} // scale = 5, precision = 4
  • 0.00001 = 1 * 10^{-5} // scale = 5, precision = 1

可以看出，在 scale 相同的情况下，precision 越大能够表示的数值越精确，这也符合 precision 英文含义。
一个特殊的场景是与 0 相等的数值（比如 0.000），它们的 precision 总是等于 1 。

Scale

对于 scale，如果它是零或正数，表示小数点右边数字的数量；如果是负数，则 unscaledValue 将乘以 10^{-scale} 。比如如果 scale 的值为 -3 ，那么 unscaledValue 的值将乘以 1000。
即 BigDecimal 的值就是将对象中的 unscaledValue 乘以 10^{-scale} 。
以下是一些相同 precision 情况下（unscaledValue 也相同）不同 scale 的例子：

  • 对于 unscaledValue = 12345， 在 scale = 5 的情况下是 0.12345；
  • 对于 unscaledValue = 12345， 在 scale = 4 的情况下是 1.2345；
  • 对于 unscaledValue = 12345， 在 scale = 0 的情况下是 12345；
  • 对于 unscaledValue = 12345， 在 scale = -1 的情况下是 123450；

但对于值相同的情况下，在设置不同 scale 的时候，其 precision 会自动变化：

  • 对于 12345， 在 scale = 5 的情况下是 12345.00000，其 precision 是 10 ；
  • 对于 12345， 在 scale = 4 的情况下是 12345.0000，其 precision 是 9；
  • 对于 12345， 在 scale = 0 的情况下是 12345，其 precision 是 5；
  • 对于 12345， 在 scale = -1 的情况下是 1234 * 10^{1}，其 precision 是 4 ；

设置 scale

使用 BigDecimal 的 .setScale 方法，可以给一个既定 BigDecimal 值设置尺度，同时会影响它的 precision 。
比如对于一个值为 12345.6789 的 BigDecimal 对象 x ，设置不同的 scale 可能导致不同的结果：

  • x = x.setScale(1, BigDecimal.ROUND_CEILING); // value = 12345.7, scale = 1, precision = 6；
  • x = x.setScale(5); // value = 12345.67890, scale = 5, precision = 10；
  • x = x.setScale(-1, BigDecimal.ROUND_CEILING); // value = 1.235E+4, scale = -1, precision = 4；
    其中设置的 scale 比当前小时，需要添加第二个参数，表示舍弃时的方式；

设置 precision

使用 BigDecimal 的构造方法，在第二个参数中使用 MathContext 类，可以指定创建对象最大允许的 precision ，同时可能会影响它的 scale 。
比如用字符串值 12345.6789 创建 BigDecimal 对象 x ，设置不同的 MathContext 可能导致不同的结果：

  • x = new BigDecimal(“12345.6789”, new MathContext(9)); // value = 12345.6789, scale = 4, precision = 9
  • x = new BigDecimal(“12345.6789”, new MathContext(8)); // value = 12345.679, scale = 3, precision = 8
  • x = new BigDecimal(“12345.6789”, new MathContext(5)); // value = 12345.6789, scale = 0, precision = 5
  • x = new BigDecimal(“12345.6789”, new MathContext(2)); // value = 1.2E+4, scale = -3, precision = 2
  • x = new BigDecimal(“12345.6789”, new MathContext(19)); // value = 12345.6789, scale = 4, precision = 9
    可以看出，对于设置的 MathContext 如果大于当前的 precision ，则不会造成影响。

总结

针对一个 BigDecimal 方法，我们可以使用 scale 和 precision 属性理解它的尺度和精度。
同时可以使用 .setScale 和 构造方法控制其 scale 和 precision 。

这个参数在实际用途是控制分割模式被使用的次数，最终效果会影响结果数组的长度。
如果 limit 比 0 大，则表示分割模式最多被使用 n 次，因此结果数组的长度不会大于 n，且数组的最后一项将包含最后一个匹配到的分割符后的所有内容。比如对于 "1,2,3,4,5," ，如果使用 split(",", 3) ，得到的结果会是 ["1","2","3,4,5,"]。

如果 limit 是一个负数，则模式将被尽可能多次地被匹配，因此数组可能有任意长度。比如对于 "1,2,3,4,5," ，如果使用 split(",", -1) ，得到的结果会是 ["1","2","3","4","5",""]。

如果 limit 是 0 ，模式也会尽可能多次地被匹配，但最后结尾的空字符串会被丢弃。比如对于 "1,2,3,4,5," ，如果使用 split(",", 0) 或 split(",")，得到的结果会是 ["1","2","3","4","5"]。
