使用TensorRT-7.2.1时遇到的坑

背景

初入 TensorRT 的坑，不太熟练，在使用过程中，遇到抛模型的构建版本与当前运行环境版本不一致。

经过排查，发现是系统所装的 TensorRT 是 7.2.1 的，而 Python 包安装的 nvidia-tensorrt 是 7.2.3 的，通过查看系统软件和Python包的命令可以检查：

解决方案

直接的解决方案是将 Python 包中的 nvidia-tensorrt 换成 7.2.1，但是安装过程中会抛找不到对应版本的错误：

在官网查找后发现，原来 7.2.1 只支持 Python 3.6 版本，而当前使用的是 Python 3.8，因此无法找到对应安装包。

需要重新使用一个 Python 环境，设定预置 Python 版本为 3.6。

尝试直接安装：

然而不带版本号却能装，装的是 7.2.3。
机智地通过装 7.2.3 的链接找到了对应 .whl 文件的 仓库地址。到对应的目录下找找看，原来是它版本号需要4位，如果写全了 7.2.1.6 是可以安装的。

后续对应安装 pycuda、opencv-python 和 torch 相关的包即可。

总结

使用 TensortRT 时可能涉及到两个环境，构建模型使用的可能是系统的版本，而运行时有可能使用的是 Python 包，需要确保两者统一。

比如：

但是，如果遇到月末的情况，就非常尴尬：

这是由于 strtotime 执行的月份操作是直接添加，比如对于 2021-03-29 的 -1 month 得到的是 2021-02-29，然而并没有这个日期，就被直接进行转换变成了 2021-03-01；同理，对于 2021-03-31 的 +1 month 得到的是 2021-04-31，同样并没有这个日期，就被直接进行转换变成了 2021-05-01。

浏览器SetCookie动作失败的原因及解决

背景

在开发过程中遇到一个奇怪的现象，登录接口明明成功了，但紧接着下一个接口却返回未登录。
于是页面上的逻辑变得非常恶心，刚刚登录成功却又被强制跳转回登录页面：

原因

经过问题排查发现，登录接口的 Response Header 里带上了 set-cookie 信息，然而在浏览器后续发送的请求中，在 Request Header 里却并没有 Cookie 信息。问题大概锁定在了浏览器收到了带 set-cookie 字段的 Response Header 却没有为当前域名的后续请求添加上关联的 Cookie。

这个问题很奇怪，仔细检查登录接口的 Response Headers 信息，可以看到一行设置。

The Set-Cookie didn’t specify a “SameSite” attribute and was defaulted to “SameSite=Lax”, and was blocked because it came from a cross-site response which was not the response to a top-level navigation. The Set-Cookie had to have been set with “SameSite=None” to enable cross-site usage.

在跨域的场景下，Set-Cookie 信息还需要指定 SameSite 字段为 None，并且开启 Secure=True操作，才能够满足跨域场景的需求。

如果后台使用的是 Laravel 框架，可以参考 这篇文章 ，修改 config/session.php 中的配置完成：

将 .env 文件中的 SESSION_SECURE_COOKIE 的值设置为 true。

总结

faiss-gpu智障的patch版本升级

今天遇到一个智障的问题，跑一个示例项目的代码过程中，抛出了 fassi 相关的错误：

拿这个错误信息去搜索，完全找不到相关的内容。

由于 fassi 自身的安装有点混乱，有时候成功安装的版本也可能缺少相关依赖库。在这个 Issue 下作者甚至直接表示仅支持通过 conda 安装，使用 pip 安装的可能有问题，现在不再支持 pip 安装了：

@sree5472 As @Silent-Zebra mentioned, the supported way of installing Faiss is through conda, and we do not support pip at the moment.

于是在这种背景下，很容易让人以为，并不是包的问题而是自己安装的有问题（开发者是没有错的，错的就是你打开的方式不正确）。
然而花了大量时间控制变量地遍历所有安装方式，包括 {pip/conda} 组合 {faiss-cpu/faiss-gpu} 等情况， 依然无法解决上述有问题。

解决方案

忙活了一个多小时，最后灵机一动，降个版本试试，当前 faiss-gpu 是 1.6.5，降到 1.6.4：

然后就不抛错了……MDZZ。。。

下载 faiss-gpu源码，切换到 v1.6.4 的标签下：

搜索关键词 cast_integer_to_long_ptr 是能够找到内容的，在 swigfaiss.swig 文件中有定义该方法，且 pytorch_tensors.py 的 swig_ptr_from_LongTensor 方法中有调用示例：

如果把版本切到 1.6.5：

搜索关键词 cast_integer_to_long_ptr 就啥也找不着了：

在 v1.6.5 版本中的 contrib/torch_utils.py 文件与之前的 pytorch_tensors.py 内容相似，但其中包含了 swig_ptr_from_UInt8Tensor、swig_ptr_from_FloatTensor、swig_ptr_from_IntTensor 等方法，唯独没有了 swig_ptr_from_LongTensor 方法，不知道是什么原因。

所以最终的结论就是，fassi 库自己升了一个 patch 版本，却修改了可能引发 breaking change 的内容，导致其它引用该方法的代码崩溃。
