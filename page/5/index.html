AngularJS中ng-if和ng-show的区别

在AngularJS框架中，遇到需要根据条件显示某元素时，通常可以有两种选择：使用ng-if或ng-show。这两者的区别也常见的关于是AngularJS的面试题。

概述

ng-if是通过移除或重新创建DOM树的节点来控制显示。
ng-show是通过添加预定义的.gn-hidecss样式设置display样式为none来控制显示。

Angular1模块间通信方式

在Angular中通常需要进行模块间通讯，总结常用的几种通讯方式如下。

通讯方式

  • 共享Service
  • 基于事件：自定义$broadcast及$emit
  • 在link函数中使用属性
  • 单向数据流+$onChanges
  • 使用第三方工具

共享Service

定义一个公有的服务，多个组件都导入该service并访问。相当于创建了一个全局变量。

基于事件：自定义$broadcast及$emit

Angular1提供了scope之间事件的传播机制。
使用scope.$emit可以往父级scope传播自定义事件并携带数据，
使用scope.$broadcast可以给所有子scope广播事件和数据，
使用scope.$on(eventName,callback)方法在所有需要接收到的scope中监听事件。
然而可能造成的问题是，每广播一次，整个scope下面的监听器都需要检测是否有对应的事件在监听并执行，如果scope层级较深，效率不高，代码也不容易追踪。

在link函数中使用属性

每个directive在定义的时候都有一个link函数，函数签名的第三个参数是attrs，代表在该directive上面的所有属性数组。
attrs提供了一些方法，比较有用的是$set和$observe，前者可以自定义attr或修改已经有的attr的值，后者可以监听到该值的变化。
利用这种方式，我们可以让在位于同一个dom元素上的两个directive进行通讯，因为它们之间共享一个attrs数组。

单向数据流+$onChanges

通过在component中声明bindings，可以进行数据的单向绑定，实现父组件与子组件之间的通信。

使用第三方工具

如Rxjs等工具。

分析

个人感觉可以将通讯场景分为两种：

  • 父子模块间的通信
  • 平行模块间的通信

对于父子模块间的通信，通常可使用事件、属性传递、单向数据流等基础方式进行，缺点是当层次较多时，需要一层层传递，比较麻烦。
对于平行模块间的通信，最常用的可用共享Service，这相当于定义了一个全局变量，缺点在于增加了不同模块之间的耦合性。
此外还可以使用第三方组件（如Rxjs）进行监听和订阅，缺点在于不容易跟踪代码，此外需要注意取消订阅，否则可能会有内存泄露的问题。

Laravel Database Query Builder insert的效率问题

背景

最近在写代码的过程中发现一个奇怪的问题，在使用Laravel的Query Builder进行数据库数据插入时，似乎当数组长度较大时，插入过程会异常缓慢。

在Laravel项目中，使用QueryBuilder的insert($values)方法时，参数$values可以是一个表示为数据库表项的数据；也可以是一个数组，包含多个数据库表项。因此，在执行多次数据插入时，可以考虑将待插入数据拼接成一个数组，只调用一次insert()操作，避免频繁的数据库IO。然而物极必反，当$values数组长度过长时，程序执行的速度变得非常缓慢，这个奇妙的现象吸引了我的注意，开始逐步分析原因。

Model::insert()方法实质是调用了Query Builder中定义的方法，仔细逐步追踪insert()方法的操作步骤，如下：

深入剖析

在这段代码中，实质有效操作语句只有3部分：

  • 1.对参数进行检查，如果为空直接返回；如果不是数组，将其封装为数组（这就是为什么参数即可以单项也可以多项的原理）。

  • 2.如果是既定数组，将数组中每一项按照其内部键值进行排序。

  • 3.调用数据库链接，执行插入操作。

通过二分法Debug，发现造成时间开销巨大的步骤出现在语句$this->cleanBindings(Arr::flatten($values, 1))中。
再进一步，是出现在Arr::flatten($values, 1)中。
打开src/Illuminate/Support/Arr.php文件，找到对应函数如下：

重点来了，这个函数完成的功能主要是：

  • 1.使用array_reduce函数，对参数$array（也就是处理后的$values）数组使用回调函数，迭代地对数组每个元素进行处理。
  • 2.针对每个元素，这里也就是$item，首先如果是一个Collection，使用->all()方法获取其值集合。
  • 3.之后根据条件判断，由于我们传入的$values是一个数组，参数$depth为1，因此将进入第二个条件语句中。
  • 4.执行return array_merge($result, array_values($item));语句。

问题就出在这个array_merge()！

我们知道，在PHP中array_merge()将一个或多个数组的单元合并起来，一个数组中的值附加在前一个数组的后面，返回作为结果的数组。
函数array_merge()底层操作的效果是，如果输入的数组中有相同的字符串键名，则该键名后面的值将覆盖前一个值；如果数组包含数字键名，后面的值将不会覆盖原来的值，而是附加到后面。
对于我们目前的情况，传入的数组都是数字键名类型的，因此键值不会覆盖，会逐个添加到数组后面。

假设$result的长度为n，假设array_values($item)的长度固定为1，则每执行一次array_merge()的时间复杂度为n
而$result的值是不断增长的，我们假设数组$value的长度为n，则$result的长度在遍历的过程中是从1增加到n。
总的时间复杂度是O(n^2)！

回到原始操作中，如果我们使用for循环逐个插入数据，时间复杂度仅为O(n)。

结论

当使用insert()方法向数据库插入数据时，如果需要插入n条数据，有两种方式：

  • 方法1：将这n条数据拼成一个数组，将数据作为参数调用一次insert()方法
  • 方法2：使用一个循环，调用n次insert()方法，即调用n次数据库插入请求，每次插入1条数据

方法1在数据处理的时间复杂度是O(n^2)，网络请求的复杂度是O(1)；
方法2在数据处理的时间复杂度是O(n)，网络请求的复杂度是O(n)。

因此，根据插入数据数组的大小以及网络情况，在此可以进行一个权衡。
根据经验，当n=1000左右，可以容忍，当n>10000时，数据处理的时间复杂度将远大于网络请求的时间开销。

Git分支基本操作

本文总结关于Git操作分支的相关操作。

创建分支

使用git branch [branch_name]可以创建新分支。

使用git checkout -b [branch_name]可以创建并切换到新分支。

使用git push --set-upstream orgin [branch_name]将当前分支提交到远程服务器上。

查看分支

使用git branch可以查看所有分支，当前分支前面会标一个*符号。

使用git branch -a 可以查看远程分支

重命名分支

使用git branch -m [old_branch_name] [new_branch_name]重命名本地分支。

若希望重命名远程分支，实质是先删除远程分支，然后重命名本地分支，再重新提交一个远程分支。

删除分支

使用git branch -d [branch_name]删除对应分支

使用git push origin --delete [branch_name]删除远程分支

Git tag的使用

基本概念

Git Tag故名思义是给Git打标签，可以对某一时间点上的版本打上标签。
在发布软件版本的时候，常用于标记某个特定版本的提交，方便查找和管理。

主要命令

显示标签

git tag 显示所有标签
git tag -l 对标签名称进行特定过滤，如
git tag -l 'v1.4.2.*' 只显示v1.4.2系列的标签

新建标签

Git使用的标签有两种类型：轻量级的（lightweight）和含附注的（annotated）。
轻量级标签是指向特定提交对象的引用（类似于一个Alias，方便记忆）；含附注标签实际上是存储在仓库中的独立对象，有自身的检验和信息，包含标签名字、电子邮件地址、日期、说明等，同时还允许使用GNU Privacy Guard(GPG)来签署或验证。
通常建议使用含附注标签，以便保留相关信息。

轻量级标签

轻量级标签实际上就是一个保存着对应提交对象的校验和信息的文件。创建轻量级标签时，不需要相关选项，直接给出标签名答即可：

php7新特性中的一些语法糖

php7相比于php5有许多新特性，通过常用的语法糖，能够使撰写的代码更加优雅、简洁。

??运算符

??运算符称作null合并运算符。若运算符的左操作数不为null，则返回左操作数；否则返回右操作数。
举例而言：

若$b不为null则$a的值为$b，否则$a被赋值字符串”default”。

使用??运算符能够使代码更加简洁。

太空船操作符<=>

太空船操作符用于比较两个表达式，当运算符的左操作数小于、等于或大于右操作数时，分别返回-1、0或1

此前一些自定义排序函数sort中的代码可以更加简洁了。

通过define()定义常量数组

Array类型的常量可以通过define()来定义了，这对于一些全局通用固定信息来说更加便捷。

匿名类

可以使用new class来声明一个匿名类，这在一些一次性使用的地方可能会比较方便。

Unicode codepoint 转译语法

可以接受一个以十六进制形式的Unicode Codepoint，打印出一个双引号或heredoc包围的UTF-8编码格式的字符串。 可以接受任何有效的 codepoint，并且开头的0是可以省略的。

Closure::call()

为unserialize()提供过滤

这个特性旨在提供更安全的方式解包不可靠的数据。它通过白名单的方式来防止潜在的代码注入。

Group use declarations

从同一 namespace 导入的类、函数和常量现在可以通过单个 use 语句 一次性导入了。

标量类型声明

此前PHP语言重要的特点就是弱类型，使代码编写者不用关心变量的具体类型，但也可能留下很多隐患。
PHP7开始支持变量类型的定义，引入一个开关指令declare(strict_type=1);
如果进行了参数和返回值的类型声明，在不开启strict_type时，PHP将尝试帮你转换成要求的类型；在开启strict_type时，如果类型不匹配就抛出错误。

注意只是在传参和返回值上不允许隐式类型转换，在使用内置函数时还是会转换，并且在变量声明的时候也不是强类型严格要求。

开源许可协议简介

开源许可协议简介

此前经常在别人的Github仓库中看到采用某某开源许可协议，这里进行总结概括，使对开源许可协议的理解更加清晰。

通常来讲，开源界5大许可协议包括：

  • GPL
  • LGPL
  • BSD
  • MIT
  • Apache

此外还有Mozilla License等协议。

GPL

GPL(GNU General Public License)，称为GNU通用公共授权许可证。
GPL是鼓励开源自由的，因此它唯一的限制是，如果软件产品中包含或修改了使用GPL的代码，那么项目必须是GPL协议。
这一点主要约束了将免费代码进行修改后闭源的谋利。
GPL协议的约束可以概括为：自由，并且使用了该产品的软件也必须是GPL协议。

由于GPL严格要求使用了GPL类库的软件产品必须使用GPL协议，对于使用GPL协议的开源代码，商业软件或者对代码有保密要求的部门就不适合集成/采用作为类库和二次开发的基础。

LGPL

LGPL(GNU Lesser General Public License)，称为GNU宽通用公共许可证。
LGPL通常用于一些(但不是全部)GNU程序库，之前曾被称为GNU库通用公共许可证，LGPL是GPL的一个为主要为类库使用设计的开源协议。和GPL要求任何使用/修改/衍生之GPL类库的的软件必须采用GPL协议不同。LGPL 允许商业软件通过类库引用(link)方式使用LGPL类库而不需要开源商业软件的代码。这使得采用LGPL协议的开源代码可以被商业软件作为类库引用并发布和销售。
LGPL协议的约束可以概括为：如果是引用包但没有使用代码，就不受GPL的限制。

公司在使用自由软件之前应该保证在LGPL或其它GPL变种的授权下，而不能是GPL。

BSD

BSD(Berkeley Software Distribution)，也称为伯克利软件发行版许可证。
BSD开源协议给于使用者很大的自由。可以自由的使用，修改源代码，也可以将修改后的代码作为开源或者专有软件再发布。当发布使用了BSD协议的代码，或则以BSD协议代码为基础做二次开发自己的产品时，需要满足三个条件：
1.如果再发布的产品中包含源代码，则在源代码中必须带有原来代码中的BSD协议。
2.如果再发布的只是二进制类库/软件，则需要在类库/软件的文档和版权声明中包含原来代码中的BSD协议。
3.不可以用开源代码的作者/机构名字和原来产品的名字做市场推广。
BSD协议的约束可以概括为：你可以自由地使用代码，只需要保留代码原有的协议。

BSD代码鼓励代码共享，但需要尊重代码作者的著作权。BSD由于允许使用者修改和重新发布代码，也允许使用或在BSD代码上开发商业软件发布和销售，因此是对商业集成很友好的协议。
很多的公司企业在选用开源产品的时候都首选BSD协议，因为可以完全控制这些第三方的代码，在必要的时候可以修改或者二次开发。

MIT

MIT是和BSD一样宽范的许可协议，作者只想保留版权，而无任何其他的限制。
只需要在发行版里包含原许可协议的声明，无论你是以二进制发布的还是以源代码发布的。
另外，相比于BSD协议更加宽松的一点是，可以使用开源代码的作者/机构名字和原来产品的名字做市场推广。

Apache

Apache Licence是著名的非盈利开源组织Apache采用的协议。该协议和BSD类似，同样鼓励代码共享和尊重原作者的著作权，同样允许代码修改，再发布（作为开源或商业软件）。需要条件和BSD类似：
1.需要给代码的用户一份Apache Licence
2.如果你修改了代码，需要再被修改的文件中说明。
3.在延伸的代码中（修改和有源代码衍生的代码中）需要带有原来代码中的协议，商标，专利声明和其他原来作者规定需要包含的说明。
4.如果再发布的产品中包含一个Notice文件，则在Notice文件中需要带有Apache Licence。你可以在Notice中增加自己的许可，但不可以表现为对Apache Licence构成更改。

Apache Licence也是对商业应用友好的许可。使用者也可以在需要的时候修改代码来满足需要并作为开源或商业产品发布/销售。

总结：通常来讲，商业使用通常闭源，因此只能选择BSD、MIT和Apache；如果只是引用，可以使用LGPL和Mozilla。
