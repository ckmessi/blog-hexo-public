使用bumpversion管理版本

bumpversion 是一个小型的命令行工具，用于简化应用发布时的版本控制。它主要能够完成以下几个功能：

  • 根据命令升级版本号；
  • 自动替换项目相关文件中的版本信息；
  • 自动创建一次 Git 提交；
  • 为这一次提交打上版本升级的标签；

安装方式

使用方式

最基础的使用方式是在命令行中使用命令：

其中 part 字段是命令的必填参数，表示要增加的版本部分。通常版本号会以 [major].[minor].[patch] 形式，比如 1.2.3 表示表示主版本号为 1，小版本号为 2，补丁版本号为 3。
使用上述 bumpversion 命令时，如果原始版本号为 1.2.3，则使用 bumpversion patch 会升级到版本 1.2.4，使用 bumpversion minor 会升级到版本 1.3.0，使用 bumpversion major 会升级到版本 2.0.0。

option 是配置参数的输入位置，可选内容有许多，具体可参考 官方文档，而 file 表示了将要修改的文件（替换文件中的版本信息），如果没有提供的话，则会查找配置文件中 [bumpversion:file:...] 列表。

配置项

所有配置信息既可以在命令行中指定，也可以通过配置文件的方式指定。通常使用 bumpversion 的项目会在根目录包含一个 .bumpversion.cfg 文件，这样在根目录执行 bumpversion 命令时，会自动根据这个文件加载相关配置。

配置项中常用的配置有以下几类：

  • 全局配置：使用 bumpversion 命令的通用配置；
  • 版本部分指定配置：关于版本形式的相关配置；
  • 文件指定配置：修改文件内容的相关配置

全局配置

全局配置主要就是以下几类，都比较实用，可以关注

  • current_version：当前版本必须指定，没有默认值；
  • new_version：新版本号，可以直接指定，如果没有指定则会根据命令自动生成；
  • tag：是否创建标签，默认为 False，用于标识项目关键版本，并在项目代码的持续集成中扮演关键作用；
  • tag_name：标签名称，默认为 v{new_version}，可根据需要自定义修改；
  • commit：是否创建提交，默认为 False，适用于 git 或 Mercurial 工具；
  • message： 在创建 commit 时的消息

版本部分指定配置

一个版本字符串通常包含一个或多个部分，比如版本 1.0.2 包含了 3 个部分，通过 . 符号分隔。默认情况下这些部分被命名为 major、minor 和 patch ，在 bumpversion 中可以使用 parse/serialize 参数自定义配置。
这部分内容不常用，通常使用遵循默认的版本升级逻辑即可。

文件指定配置

bumpversion 可以帮助开发者快速替换文件中的版本信息，避免对一个一个文件进行手动修改。

总结

bumpversion 是一款基于 Python 开发的命令行工具，使用 bumpversion 可以快速完成项目中版本的升级和管理，减少繁琐的手工操作。

  • env.py 是执行 alembic 迁移工具命令时触发的 Python 脚本代码，它包含了一些指导，比如怎么配置和生成 SQLAlchemy 引擎，如何在引擎中建立带事务的连接，如何触发迁移引擎等等。env.py 是迁移环境的一部分，它可以被各种自定义用来控制不同的迁移方式。对于大多数情况而言，最核心的作用在于它定义了 alembic 操作的到底是哪个数据库；
  • README 文件的初衷估计是为了说明如何操作，但实际效果来看比较鸡肋，并没有任何用处；
  • script.py.mako 是一个 Mako （一种 Python 模板库）模板文件，用于生成新的迁移脚本，即 versions/ 目录中的新文件。通常情况下不需要修改，但也可以自定义 upgrade() 和 downgrade() 方法模板，这样在多数据库环境中就可以扩展出 upgrade_engine1() 和 upgrade_engine2() 等方法；
  • versions 目录保存了每一个单独的版本脚本，它使用 GUID 标识每个文件。在 Alembic 中，版本脚本的顺序与脚本自身的指令相关，理论上是可能把一部分版本文件与另外一部分隔离开，允许从不同分支合并的迁移能够被手动合并、执行。

使用Mkdocs快速搭建高颜值文档

简介

一直在寻找一种撰写项目文档的最佳方式，希望使撰写文档更加便捷、查看文档更加方便，同时更好地管理文档的版本控制。
传统方案可能是使用 Word 等办公软件撰写文件，然后把文件发送给对方。这种方法有诸多不便：

  • 对方必须下载或缓存文件才能查看，造成设备空间浪费；
  • 跨平台友好性不足，对方的操作系统如果与创建者不同，则无法打开或无法编辑；
  • 文档不断修改，只能以文件名后缀 _v2、_v3 用于区分版本，相对杂乱；

现有通用的方法可以使用类似石墨文档等在线文档协同编辑工具。
这里介绍一种更编程化的方式，使用 Mkdocs 快速生成文档。

解决方案

Mkdocs 是一款快速、简洁、直接的高颜值静态页面生成器，能够快速构建项目文档。

文档源文件以 Markdown 格式撰写，通过一个简单的 YAML 文件进行配置。在撰写文档的过程中，可以实时地预览生成的文档效果。

Mkdocs通过配置文件管理主题样式，有许多成熟的样式模板可供选择，也完全可以自己构建一套样式。

由于 Mkdocs 能够生成静态页面，因此文档可以被方便地部署在 GitHub 页面、Amazon S3 或其它任何地方。
