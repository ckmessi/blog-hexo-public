Hexo配合Github Pages部署个人博客

在使用Hexo搭建个人博客系统后，很自然的需求是要将其发布到网上供读者访问。此时需要进行Hexo个人博客的部署。
常规的方案是部署到Github上。

修改_config.yml文件

命令通常是用来帮助一键完成的，这种方式简洁、快速，能够提供“隐形”的服务，使用户能够在不需要完全了解其工作原理情况下完成任务。
但是这种情况有一个弊端，即无法“自由”地控制相关操作的细节。
因此，接下来希望从原理角度探索一下命令hexo deploy的具体操作，并实现自由模式的手动部署hexo到github。

hexo deploy部署到Github上的原理

使用hexo deploy命令将博客内容部署到Github上的原理可以总结为以下三点：

  • Hexo generate能够生成public目录，这个目录中包含的都是前端静态文件，不需要服务端支持，能够直接通过浏览器访问。
  • Github的github page服务。每个Github账号有且仅有一个特殊仓库来放置个人主页，仓库名字必须是[username]/[username].github.io，通过域名http://[username].github.io/能够访问个人主页。
  • 将Hexo生成的静态网站通过git提交到Github仓库中

因此明确了这些原理，我们完全可以通过手工操作的方式，实现博客内容的部署。基本操作步骤可划分为三步：

  1. 使用hexo generate命令生成public目录（之后我们可能会将名字换为blog-hexo）
  2. 将public目录下的内容拷贝至[username].github.io在本地目录下的子目录blog-hexo
  3. 使用git commit提交

博客放置次级目录的可行性验证

但有些时候我们并不希望将博客部署到根目录[username].github.io下，因为个人主页可能包含多方面内容，如about页面，主页时间轴等，博客只是其中一项。

我们使用nginx服务器简单模拟一下静态文件的访问情况。
如果我们直接将nginx的目录指向/path/to/hexo/public/，此时使用localhost:4001/能够访问到正常的博客页面
如果我们直接将nginx的目录指向/path/to/hexo/，此时使用localhost:4001/public/能够访问到页面，但是css全乱了。

究其原因是由于hexo generate的时候，将所有js和css文件的路径设置为以/css/*.css形式，比如：

这时候关键在配置项_config.yml中的配置项root，修改其为/public/

再访问localhost:4001/public，正常显示！
可以看到index.html文件中的css文件路径都变成了：

总结

  • hexo deploy的原理是设定好Github仓库目录，自动完成提交等操作
  • 可以通过生成静态文件，并手动提交到Github仓库的方式，手动完成部署工作
  • 通过修改配置文件_config.yml的public_dir配置项，能够修改generate目录的名称
  • 通过修改配置文件_config.yml的root配置项，能够修改静态页面中css和js的引用路径前缀

手动部署的操作为：

  • hexo generate
  • cp -r blog-hexo /path/to/[username].github.io/
  • git add blog-hexo
  • git commit -m "[commit infomation]"
  • git push

随机游走

随机游走（Random Walk）是一种现象，也称为随机漫步，指的是基于过去的表现，无法预测未来的发展步骤和方向。

随机数序列和伪随机数序列的可视化

随机数与伪随机数的区别可以通过随机游走现象直观表现。
平方数取中法随机序列生成器是众多伪随机数生成器的一种，这个序列的随机性只取决于最初种子的随机性，一样的种子将导致一样的序列。
分别用随机游走把两个序列表现出来。伪随机数序列最终一定会重复，因为当算法中出现了曾经使用过的种子时，循环将开始。在伪随机序列重复之前的长度，被称为周期（Period），周期严格地由最初种子的长度所限制，比如2位数种子在周期内最多生成100个随机数。

伪随机数的实用性

如果使用足够大的种子，在重复之前，序列中的数字将会扩大到万亿个。这个关键区别很关键，当伪随机地生成数字时，将有一些序列永远无法生成。所以伪随机序列如果要和随机序列没有什么差别的话，就需要对计算机而言，尝试所有的种子并寻找一个匹配值必须是无法实现的，这导出了计算机科学中的一个重要差别，什么是可能的（Possible），以及在一个合理时间范围内什么是可能的（Possible in a reasonable amount of time）。实际安全（Practically secure）和完全安全（Perfectly Secure）。
（参考网易公开课-可汗学院-古代密码学-伪随机数生成器）

简单的随机游走示意图

用前端代码canvas画布简单地模拟了一个随机游走的Demo。
基本思路是每次游走时以当前位置为基点，在0~360度中随机生成一个方向，然后前进单位长度距离。不断重复。

Hashids的使用

1. Hashids的应用场景

最近遇到一个问题，要求不把数据库保存的自增id暴露在公开的url中，防止不怀好意的用户进行撞库、爬取数据等攻击。于是需要将简单的自增ID转换为唯一标识的字符串序列，通过字符串去查找内容，这样用户无法通过修改ID号的方式遍历所有数据。

应用场景还是很明确的，我想起了两件事。
一是在网易游戏实习时，为了不让当天的维护公告内容被用户事先得知，公告url的命名不能太有规则，不能使用户“猜到”当天的链接命名。
二是校招去搜狐面试的时候，曾被问到的一个问题就是，如何为id生成唯一的随机字符串，当时答得就不是特别好。

最初方案是想着自己用一个md5函数将id加密，然后保存到数据库中，之后通过这个md5码进行查找即可。后来想想应该会有现成的轮子，不用自己撰写，而且自己撰写有可能有极小的概率无法保证其唯一性。

后来就发现了这个工具：Hashids

它具有多种语言、多种框架的适配方式。

2. Hashids在Laravel中的使用

安装

在Laravel项目中使用composer安装：
$ composer require vinkla/hashids
在Laravel项目的config/app.php进行如下修改：
Vinkla\Hashids\HashidsServiceProvider::class
如果想使用Facade，继续添加一处：
'Hashids' => Vinkla\Hashids\Facades\Hashids::class
运行命令，会生成一个config/hashids.php文件，在其中可以进行hashids相关的配置
php artisan vendor:publish

基本使用方法

需要注意的是decode方法解析出来是一个数组，需要判断其长度来获取是否有合法的解密结果。

参数配置

在config/hashids.php文件中，可以设置盐和长度等内容

其中将length设置为10，则可以保证在ID较小的时候，看起来比较高大上（默认1位数的ID加密出来只有2个字母）。

3. Hashids的一些使用策略

在了解Hashids的使用流程后，我就产生了一个疑问，既然它能够实现双向加密解密，我是否还需要在数据库中为其保留加密后的字符串呢？
通过搜索，发现了Hashids的创建者的回答很有价值。
大概意思是，需要根据实际情况进行判断：

  1. 通过Hashids产生的Id是唯一的，不需要进行检查
  2. 最小长度参数将会为生成的Id进行补全操作。
  3. 关于database vs. on-the-fly的方式对比，在我的项目中通常是使用on-the-fly方式，因为这样比较简单。通常会将一个表的主键id进行加密，因此解密的过程是很快速的。如果为slugs（这里不太懂怎么翻译？）单独创建一列，可能需要为它创建一个索引，这样数据库需要为这个不必要的索引花费更多的管理成本。

4. Hashids的原理

在Hashids的官网上有一段说明，解析了Hashids是怎么工作的。
Hashids的工作原理很像将整数转换为十六进制，但有两点不同：

  1. 字母表不是16进制，而是62进制（默认）
  2. 字母表会通过salt字符串进行随机变换
