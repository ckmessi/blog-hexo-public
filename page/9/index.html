初识AIDL

简介

AIDL（Android Interface Definition Language），称为Android接口定义语言，目的是为了实现Android的进程间通信，尤其是涉及到多进程并发情况下的进程间通信。

宏观上讲，可以将两个需要通信的进程分别视为客户端和服务端，双方定义一份aidl文件，作为通信协议约定。
服务端需要保存xxx.aidl文件，并实现其中定义的方法；客户端需要保存xxx.aidl文件，并以此尝试与服务端建立连接，调用其相关方法。
AIDL的使用过程主要可划分为以下几个步骤：

  • 创建一份aidl文件，在其中定义接口；
  • 将进程间通信的被动端作为服务端，在服务端创建Service，实现aidl文件中的接口所定义的方法逻辑，用于处理客户端的调用请求；
  • 将主动调用端作为客户端，与服务端建立连接，并调用相关方法获取结果。

Laravel+Vue前后端分离跨域CORS的一点知识

CORS（Cross-Origin Resource Sharing）是一种允许一个网页访问其它域资源的方式。
在前后端分离的Web应用中，这个知识非常关键，它是允许前端跨域访问后台API接口的关键。

此前一直没有弄得非常明白，在网络上查询得到的答案基本都是特定情况下的解决方案，不求甚解，令人很头痛。
这次稍微弄清楚了一些。

0. CORS的概念

根据MDN官方文档中HTTP访问控制（CORS）的解说，可以了解CORS概念的基本知识。

此外，CORS标准还规定了请求的两种不同类型：简单请求和非简单请求。

在实际操作过程中，针对两种不同类型的请求，需要进行的操作也是不同的。

简单请求

某些请求不会触发 CORS 预检请求，这样的请求归纲为“简单请求”。“简单请求”的请求有以下限制：
请求方式必须是下列方法之一：

  • GET
  • HEAD
  • POST

Content-Type的值仅限于下列三者之一：

  • text/plain
  • multipart/form-data
  • application/x-www-form-urlencoded

更多限制可参考简单请求。

非简单请求

如果一个请求不是简单请求，那就是非简单请求。
非简单请求会首先执行一个类型为OPTIONS的预检请求，预检请求完成之后，发送实际请求。

这是由于axios默认的方式是application/json，并不是简单请求限制Content-Type的三个值之一，这将导致请求成为一个非简单请求。
此时，客户端将先发送一个类型为OPTIONS的预检请求，要求服务器允许正式请求。在服务器端未能正确处理预检请求时，访问将报上述错误。

假定当前POST请求的参数为变量data保存的键值对。

有几个野路子：

  1. 显示设置request中header的Content-Type为application/x-www-form-urlencoded，抛弃参数data的作用，将参数添加在url中。
    提交的数据按照key1=val1&key2=val2的方式进行编码，key和val都进行URL转码。

这种方式的原理是将Content-Type手动设置为application/x-www-form-urlencoded，以确保请求的类型是简单请求。
这个比较坑的是，参数只能手动加在url里，传入的参数data毫无用处。

  1. 将参数JSON.stringfy化

这种方式的原理与前两种类似，使用字符串作为参数，这将使Content-Type自动 为application/x-www-form-urlencoded类型，data字符串参数的格式保证了服务器端接收的正确性
这个比较坑的是，需要按格式组织好参数。

  1. 将参数转为FormData提交

这种方式的原理与上一种类似，使用FormData类作为参数，将自动设置Content-Type为multipart/form-data类型，以确保请求的类型是简单请求。
这个比较坑的是，每次都需要手动逐个添加form的key-value键值，好处是可以处理文件类型。

上述三个方案都是通过将问题转化为简单请求以完成跨域访问。

背景

在前后端分离的应用中，需要使用CORS完成跨域访问。在CORS中发送非简单请求时，前端会发一个请求方式为OPTIONS的预请求，前端只有收到服务器对这个OPTIONS请求的正确响应，才会发送正常的请求，否则将抛出跨域相关的错误。

这篇文章主要总结对Laravel中处理OPTIONS请求处理机制的探索，以及如何正确处理这类OPTIONS请求的解决方案。

问题描述

Laravel处理OPTIONS方式请求的机制是个谜。

假设我们请求的URL是http://localhost:8080/api/test，请求方式是OPTIONS。

如果存在相同URL的请求，会返回一个状态码为200的成功响应，但没有任何额外内容。

Git Fast-forward的一些常识

在Git分支合并时，通常可以选择是否采用Fast-forward形式，本文探索一下不同选择造成的差异，以及Git中关于Fast-forward相关知识。

分支单向前进的情况

假定两个分支master和develop，其中master分支保持不变，而develop在master基础上添加了若干次修改。

使用Fast-forward形式

此时使用Fast-forward形式进行合并分支，执行git merge develop或git merge develop --fast-forward

由于此时分支单向前进，develop分支中的内容相比较于master分支上增量是“纯粹”的，因此Git会默认使用Fast-forward形式。
在使用Fast-forward形式时，会自动合并提交，最后在git log的操作记录中，不会出现这一次合并操作。从图中也可以看出，Fast-forward形式就是直接将HEAD指针指向了最新提交处。

禁用Fast-forward形式

在上述情况下，依然可以选择禁用Fast-forward进行合并，此时将产生一个合并提交

在禁用Fast-forward形式时，这次合并会将本次合并新建为一个commit，能够在git log操作记录中体现此次合并操作历史。
这样能够保留原来分支记录的完整性。

分支双向前进的情况

假定两个分支master和develop，其中develop以master为基础上添加了若干次修改，同时master分支也添加了若干次修改。

此时希望将develop分支合并到master分支上，是无法使用Fast-forward形式的。
在master分支执行

此时只能执行git merge develop或git merge develop --no-ff

由于此时分支双向前进，develop分支和master分支上的变化有“交叉”，因此Git会默认禁用Fast-forward形式，Git会自动使用recursive的策略进行合并，如果有冲突则生成冲突文件，否则产生一个新的合并提交。

注意，有些版本的Git会自动完成这个合并提交，并为其添加注释为”Merge branch ‘develop’”；有些版本的Git会弹出一个Vim编辑框，提示用户输入相关注释，按:wq保存退出即可。

Fast-forward和Git Rebase

还有一个比较容易混淆和困惑的点是Fast-forward和Git Rebase的区别。

Rebase能够比Fast-forward完成更多事情。Fast-forward是很简单的一件事，而Rebase更复杂。有些时候在特殊的场景下，Rebase的效果恰好与Fast-forward相同，但并不意味着它与Fast-forward的合并是一样的。
