Grunt入门

什么是Grunt？

Javascript的任务运行器（The Javascript Task Runner）

为什么需要使用任务运行器？
自动化。当执行重复任务如压缩、编译、单元测试等的工作越少，工作就越轻松。只要通过配置Gruntfile文件，任务运行器就能帮助你以几乎零付出的代价完成这些平庸的事情。

为什么使用Grunt
Grunt的生态系统很大，并且每天还在不断增长。基于成千上百的插件可供选择，能够使用Grunt自动完成许多工作。如果没有人构建你所需要的，制作并发布自己的Grunt插件也是一件轻而易举的事情。

开始

首先安装Grunt的命令行界面grunt-cli（Grunt’s command line interface (CLI) ）

Grunt项目中通常会有两个文件：
package.json
Gruntfile 可能是Gruntfile.js或Gruntfile.coffee，用于配置或定义tasks并载入Grunt插件。

Grunt的使用习惯

传统的文件夹结构特征会包含这些文件夹：src、dist和test。

  • src文件夹（有时命名为app）包含创作的源代码库
  • dist文件夹（有时命名为build）包含发布版本，已经压缩的源代码。压缩文件中所有不必要的字符，比如空格、换行、注释，都被移除，但不影响源代码的功能。压缩源代码是很有用的，因为能够减少需要传输的数据量。
  • test文件夹包含测试项目的源代码。

开发功能库并发布新版本时，有一些基础任务是必须执行的。
比如，你希望确保你所撰写的代码达到最好的实际效果，或者说代码不会造成意外的行为。为了确保这一点，你可能需要使用一个工具JSHint。Grunt有一个官方插件grunt-contrib-jshint。尤其，你可能希望确保当你修改代码时，不会打破任何规则和实用效果。因此一个比较好的策略是在每次你执行改动时都检查代码。
为了完成这个任务，我们还会涉及到一个Grunt插件grunt-contrib-watch。当文件增加、修改、或删除时，其能够运行预定义的任务，比如grunt-contrib-jshint。
好好检查你的源代码符合最优结果不足以保证它是稳定以及不包含bug的。为了创建一个鲁棒的项目，你需要对它进行测试。有一些库可以采用，比如QUnit或Jasmine。在Grunt插件中可以使用grunt-contrib-qunit来测试代码。
当需要分发你的工作时，你希望提供一个尽可能小的版本。为了创建一个压缩版本，你需要Grunt插件如grunt-contrib-uglify。进一步，除非你正在开发的项目很小，你将代码划分到多个文件中的可能性是很大的。尽管这对于开发者来说是一个好习惯，但你希望用户只包含一个文件。因此，在压缩代码后，你还希望将源代码们合并成一个单一的文件。为了实现这个目标，你还需要Grunt插件grunt-contrib-concat。

总结一下，通常会用到以下五个Grunt插件：

  • grunt-contrib-uglify
  • grunt-contrib-qunit
  • grunt-contrib-concat
  • grunt-contrib-jshint
  • grunt-contrib-watch

总结

  • Grunt是基于Node.js的自动化任务运行器，能够帮助减轻重复性的代码工作。
  • Grunt生态系统强大，有丰富的插件可供选择
  • Grunt项目中最重要的两个文件是package.json和Gruntfile
  • Gruntfile定义了运行器的配置，由4部分组织：包装函数、任务配置、载入执行、自定义任务

Hexo个人博客使用流程

这一段时间一直在使用Hexo搭建个人博客，此前研究了Hexo配合Github-Pages进行部署、Hexo添加图片、Hexo进行版本控制等相关内容。这篇文章主要总结一下Hexo个人博客的整体流程使用规范。

首先，我们把个人博客部分划分为三部分：

  1. 【仓库1】Hexo源文件，保存着博客内容的原生内容及Hexo项目自身的配置信息
  2. 【仓库2】Hexo生成文件，保存着根据主题模板生成的静态网页内容。
  3. 【仓库3】Github-Pages个人主页，用于使用[username].github.io域名访问

其中【仓库3】中可能不只包含【仓库2】的内容，使用Git的submodule方法添加【仓库2】作为其子目录。

基于上述划分，我们可以得出个人博客使用时的基本流程：

  1. 撰写Hexo源文件，完成后提交到【仓库1】
  2. 在确认生成内容无误后，使用Hexo generate生成public目录（可修改名称），提交至【仓库2】
  3. 在【仓库3】中的【仓库2】目录下拉取最新内容，然后合并到【仓库3】中，提交。

此时【仓库3】中的【仓库2】内容保存着最新生成的静态网页内容，可使用[username].github.io/[blog-project]域名访问到。

图片位置的比较

通常来讲，添加图片可以有两种方式。一是将图片上传到第三方云平台，如七牛云、新浪图床等，然后在正文中直接使用绝对路径引用；二是将图片内容保存在与文章所在项目的相关目录下，通过相对路径引用完成。

经过调研，Hexo对于两种方式都支持。

hexo deploy部署到Github上的原理

使用hexo deploy命令将博客内容部署到Github上的原理可以总结为以下三点：

  • Hexo generate能够生成public目录，这个目录中包含的都是前端静态文件，不需要服务端支持，能够直接通过浏览器访问。
  • Github的github page服务。每个Github账号有且仅有一个特殊仓库来放置个人主页，仓库名字必须是[username]/[username].github.io，通过域名http://[username].github.io/能够访问个人主页。
  • 将Hexo生成的静态网站通过git提交到Github仓库中

因此明确了这些原理，我们完全可以通过手工操作的方式，实现博客内容的部署。基本操作步骤可划分为三步：

  1. 使用hexo generate命令生成public目录（之后我们可能会将名字换为blog-hexo）
  2. 将public目录下的内容拷贝至[username].github.io在本地目录下的子目录blog-hexo
  3. 使用git commit提交

博客放置次级目录的可行性验证

但有些时候我们并不希望将博客部署到根目录[username].github.io下，因为个人主页可能包含多方面内容，如about页面，主页时间轴等，博客只是其中一项。

  • hexo deploy的原理是设定好Github仓库目录，自动完成提交等操作
  • 可以通过生成静态文件，并手动提交到Github仓库的方式，手动完成部署工作

随机游走

随机游走（Random Walk）是一种现象，也称为随机漫步，指的是基于过去的表现，无法预测未来的发展步骤和方向。

随机数序列和伪随机数序列的可视化

随机数与伪随机数的区别可以通过随机游走现象直观表现。
平方数取中法随机序列生成器是众多伪随机数生成器的一种，这个序列的随机性只取决于最初种子的随机性，一样的种子将导致一样的序列。
分别用随机游走把两个序列表现出来。伪随机数序列最终一定会重复，因为当算法中出现了曾经使用过的种子时，循环将开始。在伪随机序列重复之前的长度，被称为周期（Period），周期严格地由最初种子的长度所限制，比如2位数种子在周期内最多生成100个随机数。

伪随机数的实用性

如果使用足够大的种子，在重复之前，序列中的数字将会扩大到万亿个。这个关键区别很关键，当伪随机地生成数字时，将有一些序列永远无法生成。所以伪随机序列如果要和随机序列没有什么差别的话，就需要对计算机而言，尝试所有的种子并寻找一个匹配值必须是无法实现的，这导出了计算机科学中的一个重要差别，什么是可能的（Possible），以及在一个合理时间范围内什么是可能的（Possible in a reasonable amount of time）。实际安全（Practically secure）和完全安全（Perfectly Secure）。
（参考网易公开课-可汗学院-古代密码学-伪随机数生成器）

基本思路是每次游走时以当前位置为基点，在0~360度中随机生成一个方向，然后前进单位长度距离。不断重复。
