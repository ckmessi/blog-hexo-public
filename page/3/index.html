一次 Swagger-UI 3.30.1 升级引发的故障

现象

使用 fastapi 框架搭建的后台服务启动后，访问 /docs 文档地址显示正确，但无法使用，现象如下：
1、点击任意一个分组下的接口，会自动跳到 Default 组下；
2、点击 Default 组下的任何接口，在展开之后内容一直停留在无限加载中的状态；

查找原因

在服务器和本地都复现了，由于近期并没有版本更新，初步判断是依赖的第三方组件升级导致的问题。
由于 \docs 页面是基于 Swagger-UI 渲染的，在其 GitHub 项目主页中找到相关线索。
在 Issue 页面，12个小时前有人报了这个 Issue，描述的现象与我们遭遇的基本一致。

这个问题暂时没有得到修复。
评论区中，立刻有人将这个问题报到了我们所使用的 fastapi 项目中：

而 fastapi 框架的作者 tiangolo 也立刻进行了回复，并对 fastapi 的源码进行了 更新，使其暂时固定依赖于 swagger-ui@3.30.0 版本。

Pin Swagger UI while this is fixed: swagger-api/swagger-ui#6249

解决方案

最优的解决方案自然是等待 Swagger-UI 的作者完成修复或版本回滚，毕竟仅是 3.30.0->3.30.1 的 patch 发布，不应该有 breaking change 存在，更何况直接影响到所有依赖它的第三方库。
在问题被解决之前，如果需要临时快速修复问题，可以参考 tiangolo 的方法，直接修改 fastapi 的源码。
步骤示例：
1、在 Docker 镜像中查找 Python 环境文件所在，一般是 /usr/lib/local/python3/site-packages/fastapi；
2、修改其中的 openapi/docs.py 文件，将定义的swagger_js_url 和 swagger_css_url 的链接中 swagger-ui-dist@3 改为 swagger-ui-dist@3.30.0；
3、重启 docker 容器（重启基于 fastapi 的服务）；
4、刷新网页

这种方式可以快速临时恢复工作环境，避免开发工作卡在这个问题上。

后续

快下班时又搂了一眼 Swagger-UI 的 Issue，发现另一个第三方库也遭灾了，该作者也选择了相同的方式固定 swagger-ui 的版本号。

参考链接

  • https://github.com/swagger-api/swagger-ui/issues/6249
  • https://github.com/tiangolo/fastapi/pull/1763

处理Windows编辑的文件在Linux系统中多出^M的办法

背景

项目中添加了一个 .sh 脚本文件，在 Windows 系统中编辑后上传到 Git 服务器，在 Linux 环境中使用时发现 Shell 抛奇奇怪怪的错。
经过排查后确定，是由于 Windows 系统中编辑的文件换行符与 Unix 系统不同导致的。
通过 cat -v [filename] 可以查看到文件内容的末尾带有 ^M 字符。
这个字符在普通代码文件中不会有影响，但对于 Shell 脚本而言会造成错误。

解决方案

使用 sed 命令直接替换文件内容：

特别需要注意的是，^M 字符显示出来是两个字符 ^ 和 M 的组合，但上述替换中不能直接输入 ^M，那样会查找到普通的字符 M，正确的输入方式是 Ctrl + V，然后 Ctrl + M。
完成替换后，可以使用 cat -v [filename] 再次查看，观察文件末尾的 ^M 是否已经去除。

网络上搜索的方案通常还会提到使用 tr -d 命令进行去除。
