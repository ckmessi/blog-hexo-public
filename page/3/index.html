使用SSH搭建端口转发的两个应用场景

在实际场景中，在需要访问内网地址时，可能会遇到两个场景：

  1. 需要在本机访问内网某个IP+端口；
  2. 需要在公网访问内网某个IP+端口；

需要注意的是，无论哪种方式，都至少需要一台 中转服务器 已经具备了访问 内网服务器 的SSH通道。
即在某认情况下，能够通过这台 中转服务器 SSH连接到 内网服务器 上：

明确几个概念：

  • 本机 ：操作者当前使用的机器；
  • 中转服务器 ：具有公网访问权限的一台云服务器；
  • 内网服务器 ：能够通过 中转服务器 SSH转发连接上的一台内网服务器；
  • 内网目标服务器 ：在内网中，不能直接访问，但可以通过 内网服务器 访问到的目标服务器；

场景1：在本机访问内网IP+端口

比如，在内网通过访问 http://192.168.1.11:3333 能够访问 Git 仓库所在的 Gitlab 网址。若希望在 本机 也能够访问 Gitlab 网址，则可以使用SSH建立一个端口转发。
在这种应用场景下，只需要搭建 目标内网服务器IP+端口 到 本机 的端口转发即可，但由于 本机 无法直接连接 内网目标服务器 ，因此需要通过 中转服务器 及 内网服务器 作为跳板机桥梁，使用上中转服务器的SSH通道转发。实际上一条命令进行了两次端口转发。

场景2：在公网访问内网IP+端口

比如，需要使其它人能够在公网访问内网 192.168.1.20 服务器的 12308 端口。这时候需要借用 中转服务器，假定其IP地址为 cloud-server-ip ，使其它人在公网访问 cloud-server-ip:12308 时，实际访问的是内网 192.168.1.20:12308 的内容。
在这种应用场景下，通常需要搭建 目标内网IP+端口 到 中转服务器 的端口转发。
此时使用方式通常分为两步：

  1. 先SSH到 内网服务器 上；
  2. 在 内网服务器 上建立 内网目标服务器 至 中转服务器 的端口转发；

因此需要有两个前提条件：

  1. 能够通过 中转服务器 SSH连接到 内网服务器 上，这是之前提到的前提；

终端使用base64进行编码解码

大多数终端环境都有 base64 命令，功能是将一段文本按照特定格式进行编码。

base64 是一种基于64个可打印字符来表示二进制数据的表示方法。
由于 64 = 2^6 ，因此每6个比特位可以代表一个打印字符。3个字节有24个比特，需要使用4个base64单元来表示，即4个可打印字符能够表示3个字节。
base64编码通常用于处理文本数据的场合，能够表示、传输、存储一些二进制数据。
完整的base64定义可见 RFC 1421 和 RFC 2045。

初识Drools

引言

Drools 是由 JBoss 公司开发的一款基于 Java 语言的开源规则引擎。
规则引擎的用途及价值不再赘述，规则引擎能够使用预定义的语义模块编写业务决策，使业务决策与应用程序代码分离，方便灵活配置。Drools 是一套业务规则管理系统（Business Rules Managemant System，BRMS）解决方案。它不仅提供了核心的业务规则引擎（Business Rules Engine，BRE），还具有页面身份认证及规则管理应用等其他关联支持。

Drools 的组成

Drools 可以视为一套规则管理系统的生态系统，其由以下几个部分组成：

  • Drools Workbench，提供用于身份验证及规则管理的用户交互界面；
  • Expert，核心业务规则引擎，是 Drools 的核心和前身，能够通过 Rete 算法实现模式匹配；
  • jBPM，工作流的集成；
  • Fusion，解决复杂事件处理（Complex Event Processing，CEP）的特性；
  • Planner，自动化规划器；

具体详情可以参考 Drools 的官方文档 。

有些场景下，我们希望将一些操作的集合组合成一个事务，即要么失败，要么全部成功。

这样，在方法1 function1 执行过程中，如果抛出了异常，则在 function1 中数据库操作将会回滚。

但是，仅仅按照上述代码进行调用，可能不会生效，有可能是因为未通过接口方法调用的情况导致的。

总结了一些 @Transactional 是否生效的情况：

  • @Transactional 加于 private 的方法，无效
  • @Transactional 加于未加入接口的 public 方法，再通过普通接口方法调用，无效。
  • @Transactional 加于接口方法，无论下面调用的是 private 或 public 方法，有效。
  • @Transactional 加于接口方法后，被本类普通接口方法直接调用, 无效。
  • @Transactional 加于接口方法后，被本类普通接口方法通过接口调用，有效。
  • @Transactional 加于接口方法后，被它类的接口方法调用，有效。
  • @Transactional 加于接口方法后，被它类的私有方法调用后，有效。

总而言之， @Transactional 是否生效，仅取决于是否加载于接口方法，并且是否通过接口方法调用（而不是通过本类调用）。

如果遇到错误：

这种情况很可能是，添加了 @Transactional，但是却没有抛出异常，如果这个子方法没有将异常往上整个方法抛出，或整个方法未将异常往上抛出，那么该异常就不会触发事务进行回滚，事务就会在整个方法执行完后就会提交，这时就会造成 Transaction rolled back because it has been marked as rollback-only 的异常。
