浏览器SetCookie动作失败的原因及解决

背景

在开发过程中遇到一个奇怪的现象，登录接口明明成功了，但紧接着下一个接口却返回未登录。
于是页面上的逻辑变得非常恶心，刚刚登录成功却又被强制跳转回登录页面：

原因

经过问题排查发现，登录接口的 Response Header 里带上了 set-cookie 信息，然而在浏览器后续发送的请求中，在 Request Header 里却并没有 Cookie 信息。问题大概锁定在了浏览器收到了带 set-cookie 字段的 Response Header 却没有为当前域名的后续请求添加上关联的 Cookie。

这个问题很奇怪，仔细检查登录接口的 Response Headers 信息，可以看到一行设置。

The Set-Cookie didn’t specify a “SameSite” attribute and was defaulted to “SameSite=Lax”, and was blocked because it came from a cross-site response which was not the response to a top-level navigation. The Set-Cookie had to have been set with “SameSite=None” to enable cross-site usage.

在跨域的场景下，Set-Cookie 信息还需要指定 SameSite 字段为 None，并且开启 Secure=True操作，才能够满足跨域场景的需求。

解决方案

如果后台使用的是 Laravel 框架，可以参考 这篇文章 ，修改 config/session.php 中的配置完成：

2、修改 secure 配置项：
将 .env 文件中的 SESSION_SECURE_COOKIE 的值设置为 true。

总结

@sree5472 As @Silent-Zebra mentioned, the supported way of installing Faiss is through conda, and we do not support pip at the moment.

于是在这种背景下，很容易让人以为，并不是包的问题而是自己安装的有问题（开发者是没有错的，错的就是你打开的方式不正确）。
然而花了大量时间控制变量地遍历所有安装方式，包括 {pip/conda} 组合 {faiss-cpu/faiss-gpu} 等情况， 依然无法解决上述有问题。
