关于将COCO格式多边形绘制成分割图

背景

COCO 数据集格式的目标标注信息中的 segmentations 字段，通常可以表达一个目标的分割信息。其主要存在三种格式：Polygon、Uncompressed RLE 和 Mask，这三种格式可以互相转换。
在常见的数据集中，通常会使用 Polygon 格式，实质上是通过其外轮廓的所有点表达了这个目标对象的信息。这种存储方式非常便捷，在一些标注工具中（比如 Paper.js ）可以直接读取并编辑其中的点，修改目标的外轮廓，并重新保存。
然而在实际进行分割训练时，有可能需要将其转为掩码信息，即对应每一个像素上的类标签。
幸运的是，对 Python 编程者而言，COCO 数据集的官方 SDK 支持包 pycocotools 包含了相关的接口，可以方便地进行调用。

绘制方式

对于给定任务将一个多边形转换为掩码图，可以直接使用 pycocotools 的 mask_util 包中的相关方法：

其中 segmentations 是一个实例对象的分割信息，height 和 width 是当前图像的高宽信息。
segmentations 的格式是一个二维数组，数组中的多个元素表示当前实例对象的彼此分离的多个部分。每个元素又是一个一维数组，包含着那个部分的外轮廓点信息。
segmentations 这个数组的长度有可能是 1，也有可能是一个大于 1 的数量。大多数情况都是 1，因为较少目标会以 彼此分离的多个部分 的形式出现。
比如 segmentations 的内容可以是：segmentations = [[100, 100, 500, 100, 500, 500, 100, 500, 120, 300]]，表示一个大概在位置 (100, 100) -> (500, 500) 的目标。
比如 segmentations 的内容可以是：segmentations = [[100, 100, 200, 100, 200, 200, 100, 200, 120, 150], [300, 300, 400, 300, 400, 400, 300, 400, 320, 350]]，其中包含了两个分离的部分，前者大概在 (100, 100) -> (200, 200) 位置，后者大概在 (300, 300) -> (400, 400) 位置。
需要注意的是，如果 segmentations 数组中的元素个数多于一个，则它们彼此之间是分离的（因为如果不分离，就可以直接融合起来重新计算外轮廓点）。

前端页面突然被登出的坑

背景

在进行项目产品测试的过程中，产品经理发现一个奇怪的现象：她在一个页面上登录后，不做什么操作，过一会儿账号就被自动登出，跳转到了登录页面。
这种现象在其它人使用过程中没有发生过。
于是我好奇地展开了探索的过程。

研究

首先，这个现象的复现情况不太一致。有时候是一登录之后立刻被登出，有时候是停留页面几秒后被登出，有时候是点击某个菜单栏后出。
尝试了一下别的浏览器，没有存在这个问题，大概可以排除后端的问题。
通过 Chrome 的网络标签可以发现，在刚刚登录的时候，可以看到大部分API请求是正常返回的，HTTP状态码是200。但是一段时间后，点击页面按钮后发出的请求就返回401 NotAuthorized，即未登录账号请求的状态了。
顺着这个线索，首先思考什么情况下会造成 401 响应。正常情况下，请求能够通过服务器的身份鉴定，是因为请求的 header 中 Cookie 信息包含了相关的授权内容。
使用 Chrome 查看 Cookie，可以发现其中内容时有时无。也就是说，登录一段时间后，Cookie 就不知道被谁给清了？

这种情况非常奇怪。突然灵机一动，发现当前打开的标签页列表中，还有几个其它标签页的标题是我们的产品链接。
而在那些标签页中，都退回到了登录页面。
打开其中一个页面的控制台，可以发现它不断地在往后端发请求，但响应全是401（废话，账号被登出了，当然请求不到内容）。
然而为什么一个页面已经登出了，还在不断地发请求呢？仔细研究了一下，大概是某些页面的定时器清除逻辑没有写好，造成了这种现象。

终于真相大白！
整体的逻辑是，存在一些页面有定时发送请求的逻辑，但相关定时器的清除功能不够完善，导致页面在非登录状态（比如退出到登录页面时）依然不停地发请求。
这些请求肯定会获得 401 的响应值，前端根据相关逻辑把当前 Cookie 清空了。
由于这些页面共享一个域名，使用的 Cookie 是同一个，因此之前使用的页面就不断地被它清除 Cookie，呈现出来的状态就是不断被登出。
而之所以之前“有时候是一登录之后立刻被登出，有时候是停留页面几秒后被登出，有时候是点击某个菜单栏后出。”，是因为前端bug的存在，导致这些定时请求的频率不一样。有时候会多次发送，导致频率特别高。
唉，对于那些已经登出的标签页而言，它们的心理大概就是：我不杀伯仁，伯仁却因我而死。幽冥之中，负此良友！

解决方案

解决方案从两个角度出发考虑。
从开发角度讲，需要确保每个计时器在页面销毁之前都正确地移除了定时任务，确保账号登出后不会一直定时发送无效的 API 调用，导致返回的 401 响应清空了浏览器存储的 Cookie 信息。
在使用者角度，在使用页面的时候，保证浏览器仅打开一个当前域名下的页面，即可避免这种被登出情况的发生。

总结

从一个现实故障的角度出发，探索了奇怪现象的原因，终于找到了根源。
很舒服。不然晚上睡不着觉了。
