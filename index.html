GitHub页面上比较不同代码版本的差异

背景

在使用 Git 对项目进行代码版本控制的过程中，可能需要比较不同版本之间的差异。
然而这个功能在 GitHub 上并不好找，貌似 GitHub 页面上并没有找着比较不同版本的直接按钮或入口。

比如上图中，根本找不着哪里能够执行 比较 这个操作。

解决方案

事实上，参考着 这篇文档 可知，GitHub 是允许在分支、标记、提交、复刻、日期之间比较仓库的状态。
简单地讲，只需要修改网页 URL 即可。

比较分支、标记、提交

这三者其实是类似的，分支（Branch）使用 分支名 作为标识，标记（Tag）以 标记名 作为标识，提交（Commit）以 提交版本号SHA值 作为标识。而比较的过程将两个标识作为参数放到页面 URL 里即可。

以提交（commit）为例，在某个仓库 Repo_A 页面，比如 github.com/username/Repo_A，在这个 URL 后面加 “/compare/[commit1]..[commit2]”，表示[commit1] 和 [commit2] 的比较。比如 github.com/username/Repo_A/compare/29ea234..84ae245 表示版本 29ea234 和 84ae245 之间的差异。

另外还可以直接用符号 ^ 代替前面的一个版本，比如 github.com/username/Repo_A/compare/2c8bb74..2c8bb74^ 表示把版本 2c8bb74 和它的前一个版本比较。比较当前 main 分支与 5 个 commit 之前的差异，可以用 github.com/username/Repo_A/compare/main..main^^^^^，查看当前主分支最新提交与其往回倒数5个commit版本的差异。

其它情况也类似，比如比较分支只需要把 commit 的 SHA 值换成分支名称即可：github.com/username/Repo_A/compare/main..dev 表示 main 分支与 dev 分支的比较。

跨复刻比较

这里的复制指的是 Fork，比如从一个主仓库中 Fork 了一个仓库到个人目录，开发一段时间后希望比较与主项目的差异。
这里只需要将原来的标识前添加上 [username]:，比如 /compare/username1:[commit1]...username2:[commit2] 表示针对 github.com/username1/Repo_A 的 [commit1] 版本与 github.com/username2/Repo_A 的 [commit2] 版本的比较。注意，这两个项目需要一个项目是从另一个项目中 Fork 出来的。
另外注意，这里 URL 中的两个点 .. 变成了三个点 ...。

总结

在 GitHub 页面上，可以直接比较同一仓库或 Fork 仓库不同版本的差异。这个功能的入口不太好找，最直接的方式是在项目主页 URL 后添加 compare/ 及相关参数，即可进入比较页面查看相关内容。

参考文献

  • https://docs.github.com/en/github/committing-changes-to-your-project/viewing-and-comparing-commits/comparing-commits

使用SSH搭建端口转发需要确认GatewayPorts配置

背景

假定有三台服务器：

  • 内网服务器A，没有固定IP，能够访问外网；
  • 云服务器B，有固定IP，从服务器A和机器C都能访问到；
  • 本地机器C，开发环境。

需求是使用机器C，能够 ssh 连接到内网服务器A，进行相关的开发操作。

解决方案

在常规操作下，通过本地机器C或云服务器B，都无法主动建立与内网服务器A的 ssh 连接，唯一可能的方式，是从内网服务器A向外打通一条隧道，供外部进行访问。
而这个打通隧道的方式，是使用 ssh 命令的 -R 参数，在建立 ssh 链接的同时反向转发端口。

比如：

则从任何机器（0.0.0.0的效果）访问机器B的 20022 端口，等价于访问机器A（从机器A看到的 127.0.0.1）的 22 端口。
这样，只需要通过 ssh 命令的 -p 参数，即可建立与机器A的链接。
即上述隧道连通后，在任何机器执行 ssh -p 20022 username@ip_B 等价于执行 ssh username@ip_A（虽然这个 ip_A 是不存在的），即向机器A建立标准的 ssh 连接。

关于 GatewayPorts 的配置

在上述使用过程中，有一点可能没有达到期望的效果，即通过 ssh -R 命令建立隧道后，观察到的现象如下：

  • 在机器B上，使用 ssh -p 20022 username@127.0.0.1 能够登录到机器A；
  • 在机器B以外的机器，使用 ssh -p 20022 username@ip_B 返回错误 connection refused；

在机器B上使用 lsof 命令查看端口占用情况：

可以发现，建立的连接只绑定在了 localhost 的 20022 端口上，因此其它机器无法远程访问该端口。

出现该问题的原因，是机器B自身 sshd 服务的配置限制的，在 /etc/ssh/sshd_config 文件中，有一个配置项 GatewayPorts，它的含义大概是：

是否允许远程主机连接本地的转发端口。默认值是”no”。sshd(8) 默认将远程端口转发绑定到loopback地址。这样将阻止其它远程主机连接到转发端口。GatewayPorts 指令可以让 sshd 将远程端口转发绑定到非loopback地址，这样就可以允许远程主机连接了。”no”表示仅允许本地连接，”yes”表示强制将远程端口转发绑定到统配地址(wildcard address)，”clientspecified”表示允许客户端选择将远程端口转发绑定到哪个地址。

如果没有手动指定其为 yes，则远程端口转发将绑定在本地的loopback地址（即 localhost）上，为了实现远程端口转发的通用性，可以选择 yes 或 clientspecified。如果指定了 yes，则端口可以接受任意地址的访问；如果指定了 clientspecified，则端口绑定的地址由建立端口转发的客户端决定（即之前示例中 {ip_1} 所指定的内容。

查找原因

后来找到 某个问题 中有作者这样回复：

大意是，PosixPath 实际上就是一个 os.PathLike 的实例，除非在 Python 安装过程中某一步出问题了。可能是在 site-packages 目录中有一个 pathlib 包，导致标准的 pathlib 包不具有 PathLike 协议了？

凭着这个直觉，找了找，发现在当前环境下有两个 pathlib.py：

  • /root/anaconda3/lib/python3.8/pathlib.py
  • /root/anaconda3/lib/python3.8/site-packages/pathlib.py

两个文件内容还不一样。
把后者删除或重命名为 pathlib.py1，重新执行 torchserve 启动，就不报错了。

原因回溯

就着这个现象，与朋友讨论着查找到了 root cause。
最初版的基础镜像安装了 pathlib2，会直接装在 /path/to/env/lib/python3.x/ 下有一个pathlib.py 文件，在深一层的 /path/to/env/lib/pathon3.x/site-package 里是没有 pathlib.py 文件。
然而在当前镜像的安装过程中，因为某些未知的原因，手动又添加了一个 pathlib 库，于是在 /path/to/env/lib/pathon3.x/site-package 目录下又多了一个 pathlib.py 文件。
这就导出代码在加载的时候可能行为上不统一，某些情况加载到了 pathlib 版本1的那个文件，而那个文件的 PosixPath 并不是一个 os.PathLike 的子类。于是出现了上述问题。

解决方案

直接解决方案是删除 /path/to/env/lib/pathon3.x/site-package 目录下的 pathlib.py 文件。
更好的方法是在安装环境时，避免重复安装 libpath 和 libpath2 ，防止可能出现的冗余依赖包。

初识TorchServe

背景

TorchServe 是一个灵活易用的工具，用于将 PyTorch 模型服务化。

上述语句中的 服务化 包含了这些特性：

  • 模型仓库的构建
  • 模型部署的注册、维护和管理
  • 推理请求的处理

其中一些专用术语的解释如下：

  • 前端：处理请求/响应的组件，能够处理来自客户端的请求和响应，并管理模型的生命周期。
  • 模型工作器：负责运行实际的在模型上进行推理的操作；
  • 模型：模型可以是一个 script_module（JIT存储模型）或 eager_mode_models，提供自定义的前/后处理数据流程。模型能够从云存储或本地服务器上加载；
  • 插件：自定义的入口或授权网关，或者能够在 TorchServe 启动时载入的批量算法；
  • 模型仓库：一个放置着所有可加载模型的目录；
