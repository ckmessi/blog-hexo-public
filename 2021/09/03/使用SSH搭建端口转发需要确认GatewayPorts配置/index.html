使用SSH搭建端口转发需要确认GatewayPorts配置

背景

假定有三台服务器：

  • 内网服务器A，没有固定IP，能够访问外网；
  • 云服务器B，有固定IP，从服务器A和机器C都能访问到；
  • 本地机器C，开发环境。

需求是使用机器C，能够 ssh 连接到内网服务器A，进行相关的开发操作。

解决方案

在常规操作下，通过本地机器C或云服务器B，都无法主动建立与内网服务器A的 ssh 连接，唯一可能的方式，是从内网服务器A向外打通一条隧道，供外部进行访问。
而这个打通隧道的方式，是使用 ssh 命令的 -R 参数，在建立 ssh 链接的同时反向转发端口。

则从任何机器（0.0.0.0的效果）访问机器B的 20022 端口，等价于访问机器A（从机器A看到的 127.0.0.1）的 22 端口。
这样，只需要通过 ssh 命令的 -p 参数，即可建立与机器A的链接。
即上述隧道连通后，在任何机器执行 ssh -p 20022 username@ip_B 等价于执行 ssh username@ip_A（虽然这个 ip_A 是不存在的），即向机器A建立标准的 ssh 连接。

关于 GatewayPorts 的配置

在上述使用过程中，有一点可能没有达到期望的效果，即通过 ssh -R 命令建立隧道后，观察到的现象如下：

  • 在机器B上，使用 ssh -p 20022 username@127.0.0.1 能够登录到机器A；
  • 在机器B以外的机器，使用 ssh -p 20022 username@ip_B 返回错误 connection refused；

在机器B上使用 lsof 命令查看端口占用情况：

可以发现，建立的连接只绑定在了 localhost 的 20022 端口上，因此其它机器无法远程访问该端口。

出现该问题的原因，是机器B自身 sshd 服务的配置限制的，在 /etc/ssh/sshd_config 文件中，有一个配置项 GatewayPorts，它的含义大概是：

是否允许远程主机连接本地的转发端口。默认值是”no”。sshd(8) 默认将远程端口转发绑定到loopback地址。这样将阻止其它远程主机连接到转发端口。GatewayPorts 指令可以让 sshd 将远程端口转发绑定到非loopback地址，这样就可以允许远程主机连接了。”no”表示仅允许本地连接，”yes”表示强制将远程端口转发绑定到统配地址(wildcard address)，”clientspecified”表示允许客户端选择将远程端口转发绑定到哪个地址。

如果没有手动指定其为 yes，则远程端口转发将绑定在本地的loopback地址（即 localhost）上，为了实现远程端口转发的通用性，可以选择 yes 或 clientspecified。如果指定了 yes，则端口可以接受任意地址的访问；如果指定了 clientspecified，则端口绑定的地址由建立端口转发的客户端决定（即之前示例中 {ip_1} 所指定的内容。

总结

使用 ssh 命令的 -R 参数，可以方便地建立反向端口转发，这在希望ssh登录内网服务器或无固定IP服务器时非常有用。上述方案通常需要一台公网可访问的云服务器作为端口转发的中转，这台服务器上的 sshd 服务配置中，对于 GatewayPorts 项的配置比较重要。
