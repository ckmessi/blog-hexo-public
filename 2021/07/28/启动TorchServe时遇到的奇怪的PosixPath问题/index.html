启动TorchServe时遇到的奇怪的PosixPath问题

背景

事故发生的背景大概是这样的。在使用 TorchServe 跑通基本流程后，尝试将其部署到 docker 镜像中，然而遇到了奇怪的问题……（在我机器上明明是好好的呀？）

在执行 torchserve --start ... 命令后，屏幕中不断地抛出错误。具体的报错信息是这样的：

根据错误栈信息能够找到抛出错误的位置在 os.path.join() 这个方法可以分析，可能是第一个参数的类型有问题。

探索

搜索受阻

奇怪的是，把相关的错误放到 google 上搜索，完全找不着相关信息：

可以看出，大部分抛错的后半部分，即参数实际类型都是 NoneType 或 int 造成的，而我们遇到的是一个 PosixPath 类型的对象。

更奇怪的是，把这个报错的文件 /root/anaconda3/lib/python3.8/site-packages/ts/version.py 直接使用 Python 执行，是不会抛错的，只有当它被 TorchServe 的子线程引用的时候，会抛这个错。

同病相怜

搜索 TorchServe 的 GitHub 项目，在 Issue 中只找到了一条和我们遭遇类似的 问题，作者给出的回复很模糊，他也找不到原因，给了一个可能的原因，但很显然我们知道并不是模型相关的原因（因为目前还没有跑起模型）：

所以，其实之前报错文件中的参数是一个 pathlib.PosixPath 并没有问题，问题出在函数为什么认为这个类型不是它期待的？（理论上它是 Path 类的子类，也算是 PathLike object）。

查找原因

后来找到 某个问题 中有作者这样回复：

大意是，PosixPath 实际上就是一个 os.PathLike 的实例，除非在 Python 安装过程中某一步出问题了。可能是在 site-packages 目录中有一个 pathlib 包，导致标准的 pathlib 包不具有 PathLike 协议了？

凭着这个直觉，找了找，发现在当前环境下有两个 pathlib.py：

  • /root/anaconda3/lib/python3.8/pathlib.py
  • /root/anaconda3/lib/python3.8/site-packages/pathlib.py

两个文件内容还不一样。
把后者删除或重命名为 pathlib.py1，重新执行 torchserve 启动，就不报错了。

原因回溯

就着这个现象，与朋友讨论着查找到了 root cause。
最初版的基础镜像安装了 pathlib2，会直接装在 /path/to/env/lib/python3.x/ 下有一个pathlib.py 文件，在深一层的 /path/to/env/lib/pathon3.x/site-package 里是没有 pathlib.py 文件。
然而在当前镜像的安装过程中，因为某些未知的原因，手动又添加了一个 pathlib 库，于是在 /path/to/env/lib/pathon3.x/site-package 目录下又多了一个 pathlib.py 文件。
这就导出代码在加载的时候可能行为上不统一，某些情况加载到了 pathlib 版本1的那个文件，而那个文件的 PosixPath 并不是一个 os.PathLike 的子类。于是出现了上述问题。

解决方案

直接解决方案是删除 /path/to/env/lib/pathon3.x/site-package 目录下的 pathlib.py 文件。
更好的方法是在安装环境时，避免重复安装 libpath 和 libpath2 ，防止可能出现的冗余依赖包。
